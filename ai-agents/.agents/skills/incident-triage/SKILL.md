# Incident Triage (разбор инцидента)

Источник: адаптация набора skills `incident-*` (`incident-triage`, `incident-evidence`,
`incident-summary`, `incident-orchestrator`) автора astsyrkunov из репозитория
`ai-tools/claude-skills-market`, MR 21. Оркестратор и state-машина из оригинала
свернуты в один линейный flow под одного пользователя; имена инструментов
переведены на DWSAI-стек (`dp_sage_*`, `dp_kube_*`, `dp_gitlab_*`).

Ты разбираешь production-инцидент за один проход: триаж, сбор фактов, короткое
заключение. Глубокого RCA-расследования инфраструктуры ты не ведешь.

## Главная дисциплина

Эти правила важнее скорости и важнее красивого вывода:

- Не путай симптом, механизм сбоя и первопричину. Это три разных уровня:
  - симптом - что видно снаружи или в мониторинге (рост 5xx, latency, crashloop, рост ошибок);
  - механизм сбоя - как именно ломается (OOMKilled, исчерпан connection pool, timeout к Redis, starvation thread pool, забэклогилась очередь);
  - первопричина - почему механизм запустился (утечка соединений после деплоя, медленный SQL из-за отсутствия индекса, всплеск трафика без autoscaling).
- `OOMKilled`, `connection pool exhausted`, `timeout`, `crash loop`, `queue backlog` -
  это механизм, а не первопричина. Не выдавай их как итог, пока не подтверждена более
  глубокая причина.
- Факты до выводов: не делай заключение, под которым нет проверенного наблюдения.
- Работаешь только на чтение. Никаких мутирующих операций (рестарт, scale, rollback,
  правка конфигов) без явного подтверждения пользователя.
- Вывод для пользователя - на русском, коротко и практично. Не вываливай сырые логи
  и длинные простыни метрик.

## Источник истины

Всегда читай `.claude/context/services.yaml`. Это реестр сервисов и источник истины для:
service name, aliases, sage_group, sage_system, k8s_cluster, k8s_namespace, gitlab_project, gitlab_tenant.

Если файла нет - не выдумывай инфраструктуру. Попроси пользователя завести его по
шаблону `references/services.example.yaml` (скопировать в `.claude/context/services.yaml`
и заполнить реальными значениями) и остановись.

Если рядом есть `.claude/context/data-flows.yaml` - прочитай и его, чтобы понять, чем
сервис является в графе зависимостей:
- вероятный источник инцидента;
- пострадал от upstream-зависимости;
- показывает только downstream-симптомы.

При деградации сервиса сначала проверь его прямых соседей в data-flow, и только потом
считай его самого первопричиной.

## Параметры инцидента

До начала проверок зафиксируй (спроси у пользователя, если не задано):

- какой сервис (имя или alias из реестра);
- временное окно инцидента (`from`/`to` в ISO 8601);
- инцидент в реальном времени или исторический;
- что стало триггером (алерт, жалоба, pager, автодетект).

Историчность важна для Kubernetes: по историческому инциденту поды уже перезапускались,
их состояние не отражает момент инцидента.

## Flow разбора

Идешь по фазам линейно, без отдельного оркестратора и без файла состояния. Не
перепрыгивай к заключению, пока факты не собраны.

### Фаза 1. Триаж

1. Прочитай `.claude/context/services.yaml`, найди сервис по `name` или `aliases`.
2. Возьми `sage_group`, `sage_system`, `k8s_cluster`, `k8s_namespace`, `gitlab_project`,
   `gitlab_tenant`, `indicators_file` (если есть).
3. Запусти проверки параллельно, где можно:
   - Sage-логи за окно инцидента (см. «Sage»);
   - Sage-метрики по приоритетным индикаторам, если есть `indicators_file` (см. «Индикаторы»);
   - Kubernetes - только если инцидент в реальном времени (см. «Kubernetes»);
   - GitLab - недавние deploy-пайплайны (см. «GitLab»).
4. Сформулируй промежуточно: что наблюдаем, предполагаемый механизм сбоя, топ 1-3
   возможных причины, что проверить дальше.

### Фаза 2. Сбор фактов

Не начинай заново и не собирай лишнего. Отталкивайся от гипотез триажа и проверяй
главные 1-3 из них. Собери небольшой набор фактов, которые:

- подтверждают гипотезу;
- опровергают гипотезу;
- показывают, что пока неизвестно.

По каждому факту фиксируй: источник (Sage/Kube/GitLab), время или окно, формулировку,
какую гипотезу он подтверждает или опровергает. Не дублируй фазу 1 и не делай финального RCA.

### Фаза 3. Заключение

Сведи находки в короткую русскую сводку (формат ниже). Не выдумывай факты, которых не
нашел. Не выдавай механизм за первопричину. Дай практичные безопасные следующие шаги.

## Расследование зависимостей

Когда логи или ошибки указывают на upstream-зависимость как вероятную причину:

**Зависимость есть в `services.yaml` (наша команда)** - расследовать немедленно, не
останавливаясь. Как только в логах или ошибках выявлена внутренняя зависимость
(таймауты, HTTP-ошибки, connection refused к сервису из реестра) - до написания
финального вывода запусти для нее те же проверки параллельно:
- Sage-логи за то же окно инцидента (`sage_group`, namespace из ее конфига);
- приоритетные индикаторы из ее `indicators_file`, если есть;
- GitLab-пайплайны за окно инцидента.

Результаты включи отдельной секцией `### Зависимость: <имя сервиса>`. Если первопричина
найдена в зависимости - зафиксируй. Если и у нее есть своя зависимость из реестра -
углубляйся рекурсивно по той же схеме.

**Зависимость вне `services.yaml` (внешняя)** - остановиться:
- зафиксировать факт недоступности (таймаут, connection refused, DNS);
- указать как вероятную первопричину;
- не лезть в инфраструктуру вне зоны ответственности команды;
- примеры внешних: Kafka-кластер, сторонние HTTP-сервисы, DWH, сетевая инфраструктура.

## Sage (логи и метрики, dp_sage_*)

Инструменты: `dp_sage_logs` (AI-сводка по логам), `dp_sage_query` (сырой MageQL без сводки),
`dp_sage_completion` (промпт в MageQL). При необходимости `dp_sage_groups` / `dp_sage_systems`
для разведки, но если group и system известны из реестра - не вызывай их зря.

- `dp_sage_logs` с `group` = `sage_group` из реестра и `system` = `sage_system` из реестра (если задан), иначе namespace из `k8s_namespace`.
- Если namespace несколько - проверяй каждый отдельно.
- Всегда задавай узкое окно `from`/`to` в ISO 8601 по окну инцидента (не полагайся на
  дефолтные `hours`).
- Если окно длинное (более 2 часов) - проверь несколько коротких срезов: старт, пик, последнее.
- В `prompt` - короткий конкретный вопрос («Какие ошибки? Когда началась деградация? Кратко»).

## Kubernetes (dp_kube_*)

**Исторический инцидент (не в реальном времени) - Kubernetes не проверять.** Поды уже
перезапускались, состояние не отражает момент инцидента. Использовать только Sage.

**Инцидент в реальном времени:**

1. Получи поды для каждого кластера из `k8s_cluster` (например `dp_kube_get_pods`).
2. Проверь аномалии: высокое число рестартов, `CrashLoopBackOff`, `OOMKilled`, `Error`, `Pending`.
3. Для проблемных подов:
   - последние логи (`dp_kube_get_logs`), не более 50 строк;
   - describe пода / события (`dp_kube_describe` либо доступный аналог) с фокусом на статус и события;
   - configmap пода, если нужно уточнить конфигурацию.
4. Глубже не идти: секреты, сети, ingress не трогать.

Имена `dp_kube_*` приведены как ориентир; используй фактические dp_kube-инструменты,
доступные в твоем окружении. Все вызовы - только чтение.

## Индикаторы (метрики через Sage)

Если в реестре есть `indicators_file`, прочитай `.claude/context/{indicators_file}`. Бери из
индикаторов поле `search_request` (готовый MageQL) и группируй по приоритету:

**Priority 1 - запускать всегда** (лаги, ошибки, uptime, latency): name/description содержит
лаг, ошибк, время работы, SLA, процент ошибок, доступность, p95, p99, latency.

**Priority 2 - если P1 показал проблемы** (рестарты): name/description содержит рестарт.

**Priority 3 - только если P1/P2 проблемные** (ресурсы): name/description содержит
CPU, RAM, утилизация, тротлинг.

Для каждого выбранного индикатора вызови `dp_sage_logs` (или `dp_sage_query` для сырого
результата) с `query` = `search_request` как есть, `from`/`to` = окно инцидента,
`prompt` = «Есть ли отклонения от нормы? Кратко». Все P1-индикаторы запускай параллельно.

## GitLab (deploy-пайплайны, dp_gitlab_*)

- `dp_gitlab_pipelines` с `repository` = `gitlab_project`, `tenant` = `gitlab_tenant` -
  недавние пайплайны.
- `dp_gitlab_pipeline` - инспекция упавших или свежих deploy-джоб.
- Release-тег вида `release/YYYY-MM-DD` задает версию, а не время деплоя: тот же релиз
  могли передеплоить позже. Не используй дату из тега как время деплоя.
- Для корреляции по времени бери: время выполнения пайплайна, время выполнения deploy-джобы,
  статус джобы, целевое окружение.
- Чтобы проверить, виноват ли деплой: найди пайплайны в окне инцидента -> разбери deploy-джобы
  -> определи прод-таргет -> зафиксируй фактическое время джобы -> сопоставь со стартом инцидента.

## Цель

Дать быстрый практичный ответ:
- что сломано;
- какой механизм сбоя наиболее вероятен;
- топ 1-3 возможных первопричины;
- какие факты это подтверждают;
- что еще нужно проверить;
- какие безопасные следующие шаги имеют смысл.

## Чего не делать

- Не вести глубокое инфраструктурное расследование.
- Не выдавать финальный RCA как доказанный без фактов.
- Не путать симптом с причиной.
- Не выдавать `OOMKilled` / `connection pool exhausted` / `timeout` за первопричину.
- Не выполнять мутирующих операций без подтверждения.

## Смежные skills

- `observability-designer` - проектирование мониторинга, SLI/SLO, golden signals и
  алертов до инцидента (design-time). Этот skill - про разбор уже случившегося инцидента.
- `runbook-generator` - генерация операционного runbook по сервису (деплой, реакция на
  инцидент, обслуживание, rollback). Бери готовые шаги отсюда; разбор причины - тут.
- `work-release-manager` - проведение production-релиза и пострелизный мониторинг
  (dp_kube + dp_sage). Если инцидент связан с недавним релизом - разбор причины здесь,
  откат и пострелизные проверки - там.
