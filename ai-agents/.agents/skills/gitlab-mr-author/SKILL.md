# GitLab MR Author

Готовит **заголовок** и **описание** merge request по фактическому состоянию текущей ветки и diff.

Цель — отдать готовый к вставке в GitLab текст, который:

* имеет заголовок в формате Conventional Commits `type(scope): …` с описанием на русском;
* заполнен строго по шаблону репозитория, без отсебятины и пустых формулировок;
* опирается только на реальные изменения в diff, а не на догадки;
* честно отражает результат `project-check`, отделяя изменения текущей ветки от pre-existing drift.

## Когда использовать

Используй skill, когда пользователь готовит текущую работу к ревью: «напиши MR», «оформи merge request», «готовь к ревью», «опиши изменения для MR», «собери описание по моему diff». Триггер — наличие ветки или незакоммиченного diff, который нужно превратить в title + description для review.

## Когда не использовать

* Нужен только commit message для одного коммита — это другой контракт (одна строка Conventional Commits), не MR.
* Пользователь просит провести ревью кода, а не описать его — это `engineering-review`.
* Нет ни ветки с изменениями относительно базовой, ни незакоммиченного diff — сначала уточни, что описывать.

## Обязательные правила поведения

1. Не создавай, не пушь и не публикуй MR автоматически. По умолчанию возвращай draft (title + description) в чате. `git push`, `glab mr create`, `git commit` — только по явной команде.
2. Описывай только то, что есть в diff. Не добавляй несуществующие изменения, миграции, флаги, эндпоинты, метрики, post-deploy шаги.
3. `project-check` — обязательный шаг перед выдачей результата. Его нельзя пропустить или симулировать. Если запустить нельзя — явно скажи об этом и не помечай чеклист пройденным.
4. Заполняй каждый пункт шаблона по факту. Пустые `...` из шаблона должны быть заменены конкретикой либо явно помечены как неизвестные/неприменимые.
5. Текст описания — на русском, по правилам русской технической редактуры (см. ниже). Технические идентификаторы (имена файлов, классов, функций, API, путей, env, log events) — as-is.
6. Источник истины по шаблону — файл репозитория `.gitlab/merge_request_templates/default.md`, а не память. Всегда читай актуальный шаблон из текущего репозитория. Шаблон задаёт обязательный минимум секций; для больших и API-затрагивающих MR его дополняют секциями `## Схема изменений` (mermaid) и `## API` (см. шаг 6a), не удаляя и не переименовывая секции шаблона.

## Workflow

### 1. Определи базовую ветку и scope diff

Базовая ветка — целевая ветка MR (обычно `master` или `main`; уточни у пользователя, если не очевидно). Собери фактический набор изменений:

```bash
git rev-parse --abbrev-ref HEAD                 # текущая ветка
git merge-base HEAD origin/master               # точка ветвления
git diff --stat origin/master...HEAD            # закоммиченные изменения ветки
git diff --stat                                 # незакоммиченные изменения
git log --oneline origin/master..HEAD           # коммиты ветки
```

Используй `origin/master` (или актуальную базовую) как точку отсчёта. Если базовой ветки нет в `origin`, опирайся на локальную `master`/`main`. Прочитай ключевые места diff (`git diff origin/master...HEAD`) — заголовок и блок «На что обратить внимание» строятся из реального содержания изменений, а не из имён файлов.

### 2. Прочитай шаблон репозитория

Прочитай `.gitlab/merge_request_templates/default.md` из текущего репозитория. Это контракт структуры описания: набор секций, формулировки чеклистов, порядок блоков. Если в репозитории несколько шаблонов — выбери `default.md`, если пользователь не указал другой. Если шаблона нет — используй fallback-структуру из `references/mr_authoring.md` и явно отметь, что шаблон в репозитории не найден.

### 3. Запусти project-check (обязательно)

Запусти локальную проверку и **дождись результата**:

```bash
make project-check    # либо: just project-check
```

Если `make`/`just` недоступны или команда не определена — сообщи об этом и не помечай соответствующий пункт чеклиста пройденным. Захвати вывод: какие шаги прошли, какие упали, и на каких файлах.

### 4. Отдели свой scope от pre-existing drift

Если `project-check` падает, определи по каждому замечанию, относится ли оно к файлам из diff текущей ветки (шаг 1) или к файлам вне твоего scope (pre-existing drift — например, замечание на файл, который ты не трогал).

* Замечание на файл **из твоего diff** → это твой scope, его нужно отразить честно (проверка не пройдена) и, по возможности, поправить или назвать причину.
* Замечание на файл **вне твоего diff** → pre-existing drift; зафиксируй его отдельно как «не относится к изменениям ветки», не приписывай себе и не маскируй.

Никогда не помечай `project-check` пройденным, если он упал. Подробные правила разделения — в `references/mr_authoring.md`.

### 5. Сформируй заголовок

Заголовок — одна строка `type(scope): <описание на русском>`:

* `type` — по основному смыслу diff: `feat`, `fix`, `refactor`, `perf`, `test`, `docs`, `style`, `build`, `ci`, `chore`.
* `scope` — по затронутой области из путей файлов (например, `agent-eval`, `metrics`, `capabilities`, `mcp`), если она очевидна; иначе опусти.
* Описание — короткое, в нижнем регистре, по-русски, без точки в конце. Технические идентификаторы — as-is.

Тип выбирай по главному изменению, а не по второстепенным файлам. Детали выбора `type`/`scope` — в `references/mr_authoring.md`.

### 6. Заполни описание по шаблону

Заполни каждую секцию шаблона по факту diff и результату `project-check`:

* **Чеклист** — отмечай `[x]` только то, что действительно выполнено и проверяемо. Пункт про `project-check` отмечай по реальному результату шага 3.
* **Что и зачем изменено** — по содержанию diff: что менялось и зачем. Без воды, без пересказа имён файлов, с акцентом на наблюдаемый эффект.
* **На что обратить внимание** — отмечай только те риски, что реально присутствуют в diff (API-контракт, конфиги/env/secrets, CI/CD, миграции БД, post-deploy). Если ничего из перечисленного нет — отметь «Нет особенностей».
* **Ручная проверка** — «Не требуется» или конкретные шаги, если изменения этого требуют.
* **Связанное** — Jira/RFC/документация только если есть фактическая ссылка или явный идентификатор (например, из имени ветки или коммитов); иначе оставь плейсхолдер, не выдумывай.

Точное соответствие пунктов чеклиста фактам diff — в `references/mr_authoring.md`.

### 6a. Большие и API-затрагивающие MR — mermaid-схема и описание API (обязательно)

Если MR **большой** (несколько затронутых слоёв, новые модули/компоненты, ориентировочно от ~15 файлов или ~500 строк — это эвристика, а не жёсткий порог) **или** меняет **API-контракт** (новые/изменённые эндпоинты, схемы запроса/ответа, публичные контракты), базового шаблона мало. Дополни описание двумя секциями — не удаляй и не переименовывай секции шаблона, а добавь сверху них:

* **`## Схема изменений`** — диаграмма mermaid, отражающая реальные компоненты и поток из diff (имена — as-is из кода). Тип выбирай по сути изменения: `flowchart` для связей компонентов/слоёв, `sequenceDiagram` для потока запроса/взаимодействия, `erDiagram` для модели данных. Схема должна помогать ревьюеру понять, что и куда добавилось, а не дублировать список файлов.
* **`## API`** — структурное описание контракта по каждому затронутому эндпоинту/схеме: метод и путь, назначение, параметры/тело запроса (поля, типы, ограничения), ответ (схема, ключевые поля), и **тип изменения** (новый / backward-compatible / breaking). Для изменений схемы запроса явно укажи: новое поле опционально или обязательно, значение по умолчанию, влияет ли на обратную совместимость.

Обе секции строй только по факту diff. Не выдумывай эндпоинты, поля, компоненты и связи, которых нет в изменениях. Если MR небольшой и API не трогает — эти секции не добавляй. Подробные правила и пример — в `references/mr_authoring.md`.

### 7. Верни результат

Верни в чате:

1. **Заголовок MR** — одной строкой, готовой к вставке.
2. **Описание MR** — в блоке, готовом к вставке в GitLab, по структуре шаблона; для большого или API-затрагивающего MR — с секциями `## Схема изменений` (mermaid) и `## API` (шаг 6a).
3. **Результат `project-check`** — короткое честное резюме: прошёл/не прошёл, что упало, что относится к твоему scope, а что — pre-existing drift.

Если описывать нечего (нет diff относительно базовой и нет незакоммиченных изменений) — скажи об этом и попроси уточнить scope, не выдумывай изменения.

## Русская техническая редактура

Язык и стиль описания — по правилам skill `russian-technical-editor` (единый источник: краткие активные формулировки, без канцелярита и лишних англицизмов, идентификаторы as-is). Здесь — только MR-специфика: текст описания на русском; имена файлов, классов, функций, API, путей, env и log events — as-is.

## Ограничения

* Язык описания — русский; технические идентификаторы — as-is.
* Не выдумывай изменения, ссылки, Jira-номера, миграции, флаги и шаги проверки.
* `project-check` обязателен; его результат нельзя подменять или скрывать.
* Не помечай чеклист пройденным авансом.
* Не пушь и не создавай MR без явной команды.
* Источник структуры описания — шаблон репозитория, а не этот файл.
