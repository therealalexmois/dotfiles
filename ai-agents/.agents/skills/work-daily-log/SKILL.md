# Daily Log

Фиксирует итоги рабочей сессии в дейли-заметку Obsidian-vault. Skill - автоматическая write-сторона дейли: собирает, что реально сделано за сессию, обогащает ссылки на артефакты через MCP и дописывает блок в заметку за сегодня по схеме секций work-daily-note.

## Место в цепочке skills

- **work-daily-log** (этот skill): из сессии в одну дейли-заметку. Пишет блок `<!-- daily-log -->` со стандартными секциями.
- **work-daily-note**: ручная нормализация и чистка одной заметки. Запускается отдельно, не из этого потока.
- **work-weekly-review**: read-only консолидация дейли за неделю. Читает секции, которые пишет этот skill.

Не редактируй weekly-log, `_workspace/*`, квартальные заметки и шаблоны - у них своя структура.

## Базовые принципы

- **Только факты.** Записывай лишь то, что реально произошло в сессии: правки, MR, переходы задач, решения. Не выдумывай результаты, метрики, статусы и ссылки. Если данных нет, пропусти секцию, а не заполняй догадками.
- **Сохраняй заметку.** Frontmatter не трогай, кроме поля `updated` (его двигает скрипт). Не добавляй второй frontmatter-блок, не меняй `created` и `tags`. Не переписывай задачи пользователя в `## Tasks` и `## Meetings`, не отмечай чужие пункты сделанными.
- **Idempotent.** Весь блок daily-log живет между маркерами и переписывается целиком. При повторном запуске за тот же день не плоди дубли: прочитай уже записанный блок и отдай смерженный суперсет (старые пункты + новые), а не второй блок.
- **Скрипт для механики, модель для смысла.** Создание заметки, метки времени и вставку блока делает `scripts/daily_log.py`. Сборку текста секций и обогащение через MCP делаешь ты.

## Рабочий процесс

### 1. Подготовь заметку и контекст

```bash
python3 scripts/daily_log.py prepare --date today
```

Команда создаст заметку за сегодня из шаблона `Templates/1_Daily Planning.md`, если ее нет, и вернет JSON: `path`, `date`, `created_now`, `existing_block` (уже записанный блок daily-log или `null`), `note_body`. Если блок уже есть, ты мержишь в него, а не пишешь заново.

Дату можно задать явно: `--date 2026-06-04`. Vault по умолчанию `~/projects/work-vault`, переопределяется `--vault`.

### 2. Собери, что сделано за сессию

Пройди по контексту сессии и выдели реальные результаты:

- изменения в коде, конфиге, документации (с привязкой к задаче, если она есть);
- открытые/смерженные MR и состояние их pipeline;
- переходы задач JIRA по статусам;
- принятые архитектурные и процессные решения;
- незавершенное и блокеры, что переносится дальше;
- что узнал нового, на что обратить внимание.

Не дублируй то, что пользователь уже руками записал в `## Tasks`. Дублирующее уже записанному в `existing_block` сворачивай в один пункт.

### 3. Обогати артефакты через MCP

Для каждого артефакта подтяни заголовок и статус, чтобы ссылка читалась без открытия. См. [references/mcp-enrichment.md](references/mcp-enrichment.md) - там точные инструменты, формат ссылок и обработка ошибок. Кратко:

| Источник | MCP | Что берем | Ссылка |
|----------|-----|-----------|--------|
| JIRA | `dpJira` (`dp_jira_issue`) | summary, status | `[DWSAI-767](https://jira3.tcsbank.ru/browse/DWSAI-767)` |
| GitLab MR | `dpGitlab` (`dp_gitlab_merge-request`) | title, merge/pipeline статус | `[MR !93](https://gitlab.tcsbank.ru/.../merge_requests/93)` |
| WIKI | `dpWiki` (`dp_wixie_get`) | title | `[Заголовок](url)` |

Если MCP недоступен или артефакт не нашелся, поставь ссылку из того, что известно в сессии, и не выдумывай статус. Все write-операции в трекерах вне зоны этого skill - он только читает для обогащения.

### 4. Составь блок по схеме work-daily-note

Используй только эти заголовки (порядок сохраняй, пустые секции опускай):

```md
## Done
## In progress
## Blockers
## Decisions
## Jira
## Artifacts
## Learn
## Research
## Attention
```

Правила полей:

- **Done** - законченное за сессию, по пунктам. Активность это не результат: помечай как сделанное только реально завершенное.
- **In progress / Blockers** - незавершенное и что мешает. Это переносится дальше, не достижения.
- **Decisions** - принятые решения (высокоценный сигнал для work-weekly-review, не выдумывай).
- **Jira** - ключи `DWSAI-<n>` со ссылкой; если был переход, укажи его (`Developing -> Review`).
- **Artifacts** - по одной ссылке в строке: `<ссылка> — <статус>`, где статус из `draft / in-review / merged / shipped / abandoned / unknown`. Тип уточняй, когда полезно: MR / RFC / ADR / doc / dashboard / chat / incident / Jira issue. Разделитель ` — ` (тире) - часть контракта с work-daily-note и work-weekly-review, оставляй его как есть.
- **Learn / Research / Attention** - что узнал, что изучить, на что обратить внимание.

Текст по-русски, кратко, по делу, в стиле существующих дейли-заметок. `е` вместо `ё`.

### 5. Запиши блок

Передай собранный блок (без маркеров, их добавит скрипт) на stdin или через `--file`:

```bash
python3 scripts/daily_log.py write --date today --file /tmp/daily-block.md
```

Скрипт вставит блок перед `# Back Matter`, при повторе заменит существующий и обновит `updated`. После записи коротко отчитайся пользователю: путь к заметке и что добавлено.

## Безопасность

- Не выполняй write-операции в JIRA/GitLab/WIKI из этого skill: только чтение для обогащения.
- Не удаляй и не переписывай существующее содержимое заметки вне блока daily-log.
- Если за сессию нет содержательной работы (только чтение, обсуждение, настройка окружения, тривиальные правки) - не засоряй дейли, скажи об этом и пропусти запись.

## Пример блока

```md
## Done

- `fix(capability)` [MR !93](https://gitlab.tcsbank.ru/dwsai/dwsai-data-agent/-/merge_requests/93): гейт runtime preflight по требованиям python/libs, тесты. По [DWSAI-767](https://jira3.tcsbank.ru/browse/DWSAI-767).

## Decisions

- type-check без кэша вынесли в отдельную команду, код не трогали (flaky `unused-ignore` из теплого `.mypy_cache`).

## Jira

- [DWSAI-767](https://jira3.tcsbank.ru/browse/DWSAI-767): Developing -> Review -> RELEASE PREPARATION.

## Artifacts

- [MR !93](https://gitlab.tcsbank.ru/dwsai/dwsai-data-agent/-/merge_requests/93) — merged
- [MR !97](https://gitlab.tcsbank.ru/dwsai/dwsai-data-agent/-/merge_requests/97) — draft (на hold)
```
