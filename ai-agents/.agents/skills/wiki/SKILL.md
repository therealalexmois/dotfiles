# Wiki / Confluence через dpWiki MCP

Работа с корпоративным Confluence `https://wiki.tcsbank.ru` из агентских инструментов: поиск, чтение, выгрузка, создание и изменение страниц через `dpWiki` MCP (Spirit CLI plugin `wixie`). Аутентификацию MCP берет на себя; токены руками передавать не нужно.

## Инструменты dpWiki MCP

| Инструмент | Назначение | Тип |
|------------|------------|-----|
| `dp_wixie_get` | Прочитать страницу в stdout или скачать в файл | read |
| `dp_wixie_search` | Поиск страниц по тексту (`query`) или по `cql` | read |
| `dp_wixie_create` | Создать страницу из markdown или storage | write |
| `dp_wixie_edit` | Переименовать страницу, заменить содержимое или и то и другое | write |

Точные флаги каждого инструмента сверяй в его схеме перед вызовом, не выдумывай.

## URL и безопасность хоста

`dp_wixie_get` и `dp_wixie_edit` принимают обе обычные формы URL страницы:

```text
https://wiki.tcsbank.ru/pages/viewpage.action?pageId=5962519685
https://wiki.tcsbank.ru/display/SPACE/Page+Title
```

Для голого числового pageId собери каноничный URL: `https://wiki.tcsbank.ru/pages/viewpage.action?pageId=<PAGE_ID>`.

Передавай в `url` только адреса на `https://wiki.tcsbank.ru/` или числовой pageId. Не подставляй произвольные хосты: это может увести auth-заголовки на чужой домен. Если на входе не pageId и не URL на `wiki.tcsbank.ru`, останови и уточни у пользователя.

## Чтение страниц

По умолчанию markdown в stdout:

```text
dp_wixie_get  url=<URL>  output=stdout  format=md
```

`format`: `md` (читаемый markdown, по умолчанию), `storage` (сырой Confluence `body.storage`, лучше всего сохраняет макросы, ссылки и таблицы), `pdf`. Для последующего редактирования бери `storage`, для ответа человеку, `md`.

Скачать на диск (рекурсивно, с сохранением дерева страниц):

```text
dp_wixie_get  url=<URL>  output=file  format=md  structure  overwrite  dir=./out
dp_wixie_get  url=<URL>  output=file  format=md  structure  recursive  overwrite  dir=./out
```

Большие страницы:

1. Прочитай один раз, ищи внутри markdown заголовки и термины по вопросу пользователя.
2. Цитируй только релевантные секции, остальное пересказывай.
3. Не вываливай всю страницу обратно в чат.

Уважай копирайт и конфиденциальность: короткие выдержки, по возможности пересказ. Всегда сообщай, какие страницы использовал: заголовок и URL.

## Поиск

Свободный текст:

```text
dp_wixie_search  query="твой запрос"  limit=25  format=json
```

С ограничением по space (только вместе с `query`, не с `cql`):

```text
dp_wixie_search  query="твой запрос"  space=NMS  limit=25  format=json
```

Осторожно: на этом инстансе scoping по `space` бывает нестабилен и отдает HTTP 500 (и через `query`+`space`, и через `cql` со `space = "..."`), тогда как тот же текстовый поиск без `space` работает. Если scoped-поиск упал с 500, повтори без `space` обычным `query`, а при необходимости сузь результат по заголовку через CQL без `space` (`title ~ "..."`). Честно скажи пользователю, что выдача не ограничена спейсом, раз scoping не отработал.

Продвинутый CQL: wixie сам подставляет `type=page AND`, поэтому `type=page` не добавляй:

```text
dp_wixie_search  cql='space = "NMS" AND title ~ "Image Provisioning" ORDER BY lastModified DESC'  limit=25  format=json
```

`query` и `cql` взаимоисключающие. `format` по умолчанию `table`; для парсинга результата бери `json`.

## Создание и изменение страниц

Запись в Confluence outward-facing и затирающая: `dp_wixie_edit` заменяет содержимое страницы целиком, а не патчит. Поэтому порядок безопасный.

Создать:

```text
dp_wixie_create  space=NMS  title="Page Title"  content="<markdown>"  content-format=markdown  parent-id=<URL или pageId родителя>
```

Вместо `content` можно `file=<path>`. `content-format`: `markdown` (по умолчанию) или `storage`.

Изменить:

1. Прочитай текущее содержимое: `dp_wixie_get url=<URL> format=storage` (storage сохраняет макросы и ссылки, markdown теряет).
2. Подготовь полный новый текст страницы, а не фрагмент.
3. Покажи пользователю предлагаемое изменение или патч-сводку и получи явное подтверждение.
4. `dp_wixie_edit url=<URL> content=<полный новый текст> content-format=storage comment="что и зачем поменяли"`. Заголовок меняется через `title`; без `title` он сохраняется. Мелкую правку помечай `minor`.
5. Прочитай страницу заново и убедись, что изменения применились.

`content-format` при `edit` бери тот же, в котором готовил текст: правил storage, передавай `storage`; писал markdown, передавай `markdown`.

## Safety-правило

- Write-операции (`dp_wixie_create`, `dp_wixie_edit`) выполняй только после подтверждения в текущем шаге. По умолчанию покажи space, заголовок, родителя и суть изменения, и спроси.
- Перед `edit` всегда читай текущую версию через `get format=storage`: правка заменяет тело целиком, легко потерять контент.
- Никогда не создавай, не меняй, не переименовывай страницы без явного согласия пользователя в этом же диалоге.
- Не публикуй во внутреннюю вики секреты, токены, локальные пути и приватные данные.

## Failure modes

- Auth-ошибки или пустой результат на странице, которая должна быть доступна, попроси пользователя выполнить `dp auth login` и повтори. Токены руками не запрашивай.
- `get` вернул inline JSON-блоб макроса (например `{"mode":"list",...}`), так Confluence рендерит незнакомые макросы в markdown. Пересказывай вокруг них, не интерпретируй сырые параметры макроса.
- Пустой поиск, расширь `query`, убери `space` или перейди на CQL с поиском по фрагменту заголовка (`title ~ "..."`).
- HTTP 500 на поиске со `space` (scoping по спейсу на этом инстансе нестабилен): повтори без `space` текстовым `query`; для сужения используй CQL по заголовку без `space`. Не считай это отсутствием результатов.
- Страница вне доступных тебе space, `get` упадет; попроси доступ или верный space/pageId.
- 404 на pageId, неверный ID, недоступный space или нет прав.

## Связь с другими skill

- `work-daily-log` обогащает ссылки на вики в дейли-заметке через тот же dpWiki MCP. Этот skill про прямую работу со страницами (поиск, чтение, создание, правка), а не про логирование сессии.
- `writing-russian-editor`, источник правды по редактуре русского технического текста. Когда наполняешь или правишь русскоязычную страницу, текст готовь по его правилам, а публикацию делай отсюда.
