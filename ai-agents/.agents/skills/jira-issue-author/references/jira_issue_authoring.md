#Как оформлять технические детали Требования к оформлению задач в JIRA

Этот документ фиксирует единый подход к постановке задач, багов и эпиков в JIRA.

Документ предназначен для использования как reference в ChatGPT Sources, чтобы следующие задачи оформлялись в одном стиле: коротко, проверяемо, без додумывания фактов и без смешивания разных уровней абстракции.

---

## 1. Общие принципы

### 1.1. Язык и стиль

- Все задачи оформляются **на русском языке**.
- Формулировки должны быть **деловыми, короткими и однозначными**.
- Описание должно быть понятно:
  - разработчику;
  - тимлиду;
  - аналитику;
  - ревьюеру.
- Не использовать художественные, эмоциональные или разговорные формулировки.
- Не использовать канцелярит.
- Не писать лишние технические детали, если задача исследовательская или верхнеуровневая.
- Если задача продуктовая, заголовок должен быть **продуктовым**, а не низкоуровнево-техническим.
- Технические термины можно оставлять на английском, если они используются в проекте: `SSE`, `middleware`, `capability`, `tool`, `runtime`, `event_key`, `error_class`, `prompt`, `HITL`, `MCPHub`.

### 1.2. Информационный стиль

При оформлении задач использовать информационный стиль:

- сначала суть, потом детали;
- писать через действия, факты и проверяемые результаты;
- отделять проблему от решения;
- не превращать описание в dump обсуждения;
- перерабатывать входные данные в понятный рабочий документ.

Не использовать абстрактные формулировки без раскрытия результата:

- `улучшить`;
- `доработать`;
- `реализовать корректно`;
- `исправить логику`;
- `сделать правильно`;
- `обработка работает`;
- `повысить качество`;
- `оптимизировать`, если не указано, что именно должно измениться.

Такие формулировки допустимы только если рядом явно описан конкретный и проверяемый результат.

Хорошо:

- `Добавить обработку SendTimeoutError в SSE-стриме агента, чтобы поток завершался через RUN_ERROR без падения webserver`.
- `Убрать thread_id из labels метрик agent_model_call_duration_seconds, чтобы снизить high-cardinality в Prometheus`.
- `Добавить event_key в ручной список транзакций, чтобы связывать факт платежа с календарем платежей`.

Плохо:

- `Улучшить SSE`.
- `Доработать метрики`.
- `Исправить платежи`.
- `Сделать корректную обработку ошибок`.

### 1.3. Не додумывать факты

Нельзя добавлять в задачу факты, которых нет во входных данных.

Если факт неизвестен, нужно:

- не подменять его предположением;
- не выбирать один вариант молча;
- вынести неопределенность в `Неизвестно`;
- задать уточняющий вопрос, если без ответа задача получится недостоверной.

Допустимо формулировать осторожно:

- `требуется уточнить`;
- `если подтвердится`;
- `предположительно`;
- `нужно проверить`;
- `вариант зависит от выбранного решения`.

Недопустимо:

- добавлять неуказанные сроки;
- придумывать владельца задачи;
- придумывать API-контракт;
- придумывать названия метрик, endpoint, таблиц, флагов или ENV-переменных;
- писать тесты, которых нельзя вывести из входных данных;
- добавлять rollout, если он не нужен или не обсуждался.

### 1.4. Что важно в любой задаче

Любая задача должна отвечать на 4 вопроса:

1. **Зачем** она нужна.
2. **Что именно** нужно сделать.
3. **Как понять**, что задача завершена успешно.
4. **Как проверять** результат.

### 1.5. Требования к содержанию

- Описание должно отражать **реальный контекст**, а не абстрактную формулировку.
- Если есть архитектурные ограничения, их нужно явно указывать.
- Если есть важные ссылки на документацию, RFC, wiki, Swagger, Vault, Langfuse, Time, GitLab, DevPlatform, их нужно добавлять прямо в описание.
- Если задача является частью более крупной инициативы, это должно быть видно по формулировке.
- Если в задаче есть rollout или порядок выката, это нужно явно фиксировать.
- Если задача backend-only, frontend-only или research-only, это нужно явно указать.
- Если задача слишком большая, нужно предложить разбиение на Epic и несколько задач.

## 2. Работа с неполными входными данными

### 2.1. Когда нужно задавать вопросы

Если входных данных недостаточно, нельзя додумывать факты.

Если не хватает критичных данных, без которых задача получится недостоверной, нужно вернуть только блок `Что нужно уточнить`.

Критичные вопросы — это вопросы, которые влияют на:

- тип задачи;
- заголовок;
- scope;
- ожидаемое поведение;
- критерии успеха;
- способ проверки;
- архитектурные ограничения;
- зависимости;
- rollout;
- owner / responsible team, если это важно для постановки.

### 2.2. Какими должны быть вопросы

Вопросы должны быть:

- короткими;
- конкретными;
- связанными с результатом задачи;
- без формальных вопросов ради процесса;
- не больше необходимого минимума.

Не нужно задавать вопросы, если:

- данных хватает для минимальной рабочей постановки;
- неизвестность можно явно вынести в `Неизвестно`;
- вопрос не меняет описание задачи;
- вопрос касается реализации, которую должен выбрать исполнитель.

### 2.3. Когда можно формировать задачу без уточнений

Если данных достаточно для минимальной рабочей постановки, нужно сформировать задачу и вынести неопределенности в блок `Неизвестно`.

Пример:

```text
{panel:title=Неизвестно|borderColor=#cbd5e1}
- Нужно уточнить точное название метрики после выбора финального имени в коде.
- Нужно подтвердить, требуется ли rollout через feature flag.
{panel}
```

### 2.4. Если данных критически не хватает

Если данных не хватает для достоверной постановки, не нужно формировать финальное описание задачи.

Нужно вернуть:

```text
Тип задачи: требуется уточнение

Что нужно уточнить:
1. ...
2. ...
3. ...
```

## 3. Правила для заголовков

### 3.1. Базовые требования

Заголовок задачи должен быть:

- коротким;
- понятным без чтения полного описания;
- ориентированным на результат;
- без лишней воды;
- в одном стиле с остальным backlog.

### 3.2. Предпочтительный паттерн

Обычно используется схема:

`Глагол + объект + контекст`

Примеры:

- `Добавить интенты для поиска дашбордов`
- `Перенести системный prompt на backend`
- `Подключить MCPHub tools к агенту`
- `Упростить CI/CD после перехода на DaaS`
- `Добавить обработку таймаута в SSE-стриме агента`
- `Исправить расчет статуса платежа в календаре`
- `Обновить документацию по подключению capability pack`

### 3.3. Для продуктовых задач

Нужно использовать **понятный продуктовый смысл**, а не чисто техническую внутренность.

Хорошо:

- `Исправить отображение недоступных дашбордов в поиске`
- `Добавить форму репортинга в результатах поиска`

Хуже:

- `Исправить 403 handling в access checker`
- `Поменять boolean флаг доступности в UI`

### 3.4. Для технических задач

Можно использовать более инженерную формулировку, но она всё равно должна быть читаемой.

Хорошо:

- `Добавить RED-метрики для вызовов MCP tools`
- `Убрать thread_id из high-cardinality labels`
- `Добавить runtime-конфиг tool selector для агента`

### 3.5. Для bug-задач

Заголовок bug-задачи должен описывать видимый дефект или нарушенное поведение.

Хорошо:

- `Исправить зависание SSE-стрима при SendTimeoutError`
- `Исправить неверный статус платежа после переноса даты`
- `Исправить отображение недоступных дашбордов в поиске`

Хуже:

- `Починить баг`
- `Исправить ошибку`
- `Проблема в обработчике`
- `Пофиксить 500`

### 3.6. Для эпиков

Epic Name должен быть:

- короче обычной задачи;
- шире по смыслу;
- без лишней детализации;
- ориентирован на инициативу, а не на одну реализацию.

Примеры:

- `SOP для агента Helicopter`
- `Правила среды для агента Helicopter`
- `Сценарный HITL для агента`
- `Инструменты агента через MCPHub`
- `Capability pack для Data Agent`

## 4. Базовый шаблон для обычной задачи

Этот шаблон использовать по умолчанию для Task / Story / Technical Task.

```text
{panel:title=Контекст|borderColor=#cbd5e1}
Какую проблему хотим решить? В чем суть задачи?
Опишите текущий контекст: что сейчас не так, почему задача появилась, на кого или на что это влияет.
Зафиксируйте ограничения, если они есть.
{panel}
{panel:title=Что нужно сделать?|borderColor=#cbd5e1}
Опишите конкретные действия или этапы.
Укажите предпочтительное решение, если оно уже выбрано.
Добавьте референсы, ссылки и примеры, если они есть.
{panel}
{panel:title=Критерии успеха|borderColor=#cbd5e1}
Опишите проверяемый результат.
Каждый критерий должен быть наблюдаем через поведение системы, тест, лог, метрику, UI или документ.
{panel}
{panel:title=Тестирование|borderColor=#cbd5e1}
Опишите, как проверить результат.
Указывайте только релевантные проверки: unit, integration, e2e, smoke, manual, regression, perf.
{panel}
```

### 4.1. Как заполнять блок `Контекст`

В этом блоке нужно:

- описать текущую ситуацию;
- указать, что именно сейчас работает не так;
- объяснить, почему задача появилась;
- указать, на кого или на что это влияет;
- зафиксировать ограничения, если они есть.

Нужно избегать:

- слишком длинной исторической справки;
- реализации вместо контекста;
- воды без причинно-следственной связи;
- общих фраз без конкретной проблемы.

### 4.2. Как заполнять блок `Что нужно сделать?`

В этом блоке нужно:

- перечислить основные действия или этапы;
- указать предпочтительный путь, если он уже выбран;
- отметить, если в задаче есть исследовательская часть;
- добавить ссылки на Swagger, wiki, RFC, docs, репозитории, примеры;
- обозначить границы работы.

Если задача исследовательская — не расписывать реализацию слишком подробно.

Если задача инженерная — перечислять конкретные изменения по слоям, модулям или системам.

Если есть спорный scope — добавить блок `Не входит в scope`.

### 4.3. Как заполнять блок `Критерии успеха`

Нужно писать **проверяемый результат**, а не абстрактные намерения.

Критерий успеха должен быть наблюдаем через:

- поведение системы;
- API-ответ;
- UI;
- тест;
- лог;
- метрику;
- алерт;
- документ;
- RFC;
- dashboard;
- запись в целевой системе.

Хорошо:

- `Endpoint возвращает ответ в новом формате data/meta`.
- `Событие появляется в Statist`.
- `В Grafana доступны RED-метрики`.
- `При SendTimeoutError SSE-стрим завершается событием RUN_ERROR`.
- `Документация содержит пример подключения capability pack v1`.

Плохо:

- `Сделать хорошо`.
- `Улучшить систему`.
- `Ошибки обрабатываются корректно`.
- `Логика работает правильно`.
- `Код покрыт тестами`, если не указано, какой результат проверяют тесты.

### 4.4. Как заполнять блок `Тестирование`

Нужно указывать только релевантные проверки:

- unit tests;
- integration tests;
- e2e tests;
- smoke;
- ручная проверка;
- regression;
- perf, если нужно;
- проверка логов;
- проверка метрик;
- проверка dashboard;
- проверка документации.

Не нужно писать слишком общий текст без конкретики.

Хорошо:

- `Unit: проверить расчет статуса для просроченного, запланированного и оплаченного платежа`.
- `Integration: проверить, что SSE endpoint возвращает RUN_ERROR после ошибки в producer task`.
- `Manual: открыть dashboard в Grafana и убедиться, что метрика agent_tool_calls_total отображается с labels tool/outcome/error_class`.

Плохо:

- `Проверить руками`.
- `Написать тесты`.
- `Проверить, что работает`.
- `Прогнать регрессию`, если не указано, какую область нужно проверить.

## 5. Шаблон для bug-задач

Для дефектов используется отдельный шаблон.

```text
{panel:title=Проблема|borderColor=#cbd5e1}
Кратко опишите дефект и его влияние.
Укажите, где проявляется проблема и на кого или на что она влияет.
{panel}
{panel:title=Шаги воспроизведения|borderColor=#cbd5e1}
Опишите точные шаги, если они известны.
Шаги должны быть достаточны, чтобы другой человек мог повторить проблему.
{panel}
{panel:title=Ожидаемый результат|borderColor=#cbd5e1}
Опишите, что должно происходить.
Если известна бизнес-логика, укажите ее здесь.
{panel}
{panel:title=Актуальный результат|borderColor=#cbd5e1}
Опишите, что происходит сейчас.
Укажите разницу между ожидаемым и актуальным результатом.
{panel}
{panel:title=Дополнительная информация|borderColor=#cbd5e1}
Добавьте логи, ссылки, окружение, гипотезы, ограничения или вложения, если они есть.

Если баг касается интерфейса приложения, укажите:
- операционную систему;
- браузер и версию;
- ссылку на место воспроизведения;
- видео или скриншот, если есть.
{panel}
```

### 5.1. Правила для bug-задачи

- Заголовок должен быть **продуктовым**, а не низкоуровневым, если дефект виден пользователю.
- В шаги воспроизведения нужно писать последовательность, которую можно повторить.
- Если шаги неизвестны, это нужно явно указать.
- Если известна бизнес-логика, её стоит указывать в ожидаемом результате.
- Если дефект связан с кодом ответа API, например только `403` означает недоступность, это должно быть явно отражено.
- Если есть логи, trace id, request id, ссылки на dashboard или окружение, их нужно добавить в `Дополнительная информация`.
- Не нужно превращать bug-задачу в исследовательскую задачу, если уже понятны проблема, ожидаемый результат и способ проверки.

## 6. Шаблон для Epic

Для Epic использовать отдельный шаблон.

```text
Epic Name: <короткое название эпика>

{panel:title=Описание|borderColor=#cbd5e1}
Высокоуровневое описание области работ.
Опишите, зачем нужен Epic и какую инициативу он объединяет.
{panel}

{panel:title=Цель|borderColor=#cbd5e1}
Чётко сформулированная цель, которая должна быть достигнута по завершении Epic.
{panel}

{panel:title=Обоснование|borderColor=#cbd5e1}
Почему Epic нужен.
Укажите бизнес-требования, технические причины, проблемы текущего состояния или стратегический контекст.
{panel}

{panel:title=Преимущества и польза|borderColor=#cbd5e1}
Перечислите ключевые выгоды, которые даст выполнение Epic.
{panel}

{panel:title=Стейкхолдеры|borderColor=#cbd5e1}
Перечислите заинтересованные стороны, которые влияют на результат или зависят от него.
{panel}

{panel:title=Основные аспекты / Функциональные требования|borderColor=#cbd5e1}
Опишите основные направления работ, функции или группы задач внутри Epic.
{panel}

{panel:title=Риски и проблемы|borderColor=#cbd5e1}
Укажите потенциальные риски, ограничения, зависимости и спорные места.
{panel}

{panel:title=Критерии успеха|borderColor=#cbd5e1}
Опишите проверяемые признаки завершения Epic.
{panel}
```

### 6.1. Дополнительно для Epic

- Указывать **Epic Name** отдельно.
- Epic Name должен быть коротким, понятным и в едином стиле с остальными эпиками.
- Epic должен описывать **инициативу**, а не одну конкретную реализацию.
- В Epic лучше не расписывать низкоуровневые детали кода.
- Если Epic слишком широкий, нужно предложить декомпозицию на несколько Epic или набор задач.

### 6.2. Сокращенный шаблон для небольшого Epic

Если Epic небольшой и полный шаблон будет избыточен, можно использовать сокращенный вариант:

```text
Epic Name: <короткое название эпика>

{panel:title=Описание|borderColor=#cbd5e1}
Кратко опишите область работ и зачем нужен Epic.
{panel}

{panel:title=Цель|borderColor=#cbd5e1}
Опишите конечный результат, которого нужно достичь.
{panel}

{panel:title=Критерии успеха|borderColor=#cbd5e1}
Опишите проверяемые признаки завершения Epic.
{panel}
```

Сокращенный шаблон допустим только если:

- инициатива небольшая;
- стейкхолдеры и риски очевидны;
- нет сложного rollout;
- нет нескольких независимых направлений работ;
- нет необходимости фиксировать подробное обоснование.

## 7. Опциональные блоки

Опциональные блоки добавляются только при наличии реальной необходимости.

### 7.1. `Не входит в scope`

Использовать, если есть спорные или явно исключенные работы.

```text
{panel:title=Не входит в scope|borderColor=#cbd5e1}
- ...
{panel}
```

Примеры:

- `Frontend-изменения не входят в scope этой задачи`.
- `Изменение API-контракта не входит в scope`.
- `Production-grade enforcement token budget будет описан отдельной задачей`.

### 7.2. `Зависимости`

Использовать, если результат зависит от другой команды, сервиса, доступа, решения или внешнего артефакта.

```text
{panel:title=Зависимости|borderColor=#cbd5e1}
- ...
{panel}
```

Примеры:

- `Доступ к Vault secret для LLM Proxy`.
- `Согласованный RFC по capability pack v1`.
- `Готовый endpoint в MCPHub`.
- `Решение SRE по alert threshold`.

### 7.3. `Неизвестно`

Использовать, если есть открытые вопросы, которые не блокируют минимальную постановку.

```text
{panel:title=Неизвестно|borderColor=#cbd5e1}
- ...
{panel}
```

Примеры:

- `Нужно уточнить финальное имя метрики`.
- `Нужно подтвердить, требуется ли feature flag`.
- `Нужно проверить, есть ли готовый пример в соседней команде`.

### 7.4. `Rollout`

Использовать, если важен порядок выката.

```text
{panel:title=Rollout|borderColor=#cbd5e1}
- ...
{panel}
```

Примеры:

- `Сначала включить в dev`.
- `После проверки включить в preprod`.
- `В prod выкатывать через feature flag`.
- `После выката проверить dashboard и логи`.

### 7.5. `Ссылки`

Использовать, если есть RFC, wiki, Swagger, GitLab, Vault, DevPlatform, Langfuse, Time или другие материалы.

```text
{panel:title=Ссылки|borderColor=#cbd5e1}
- ...
{panel}
```

Не добавлять опциональные блоки ради формы.

## 8. Когда задача должна быть исследовательской

Задачу стоит делать исследовательской, если:

- ещё не выбран способ реализации;
- есть несколько архитектурных вариантов;
- важно сначала сравнить trade-off;
- результатом должен стать RFC или решение, а не сразу код;
- нужно изучить готовые решения внутри компании или снаружи;
- есть риск выбрать неверную архитектурную границу.

В таких задачах в описании нужно делать акцент на:

- исследование;
- анализ вариантов;
- trade-off;
- рекомендации;
- RFC;
- согласование решения;
- критерии выбора.

Не нужно подробно расписывать код, если он ещё не определён.

### 8.1. Критерии успеха для исследовательской задачи

Хорошие критерии успеха:

- `Подготовлен RFC с описанием вариантов, trade-off и рекомендованного решения`.
- `Зафиксированы ограничения выбранного подхода`.
- `Решение согласовано с заинтересованными командами`.
- `По итогам исследования созданы follow-up задачи на реализацию`.
- `В RFC указано, что не входит в scope первой версии`.

Плохие критерии успеха:

- `Исследование проведено`.
- `Подход выбран`.
- `Все стало понятно`.

## 9. Когда в задаче нужен RFC

Нужно добавлять этап с RFC, если:

- затрагивается архитектура;
- меняется runtime-модель агента;
- появляется новый cross-cutting layer;
- меняется policy / HITL / prompt strategy / orchestration;
- меняются границы ответственности между слоями;
- появляется новый формат артефактов;
- решение может повлиять на другие команды и сценарии.

Обычно это оформляется как пункт в `Что нужно сделать?`.

В `Критерии успеха` добавляется:

- `RFC подготовлен и согласован`;
- `В RFC описаны scope, ограничения, trade-off и критерии выбора решения`;
- `По итогам RFC созданы follow-up задачи, если они нужны`.

## 10. Как оформлять технические детали

### 10.1. Что стоит указывать

- конкретные endpoint;
- названия ENV-переменных;
- названия заголовков, например `X-Tenant-Id`, `X-Llm-Access-Token`;
- имена метрик;
- labels метрик;
- названия events;
- пути до модулей;
- названия таблиц;
- названия feature flags;
- ссылки на документацию;
- порядок rollout;
- ограничения по совместимости;
- затронутые слои архитектуры.

### 10.2. Что не стоит перегружать

- слишком длинные куски реализации;
- перечисление внутренних классов, если это не критично для постановки;
- случайные детали, которые не влияют на понимание задачи;
- фрагменты обсуждений без переработки;
- альтернативы, которые уже явно отклонены;
- детали будущей идеальной платформы, если для текущего этапа нужен простой MVP.

## 10.3. Как оформлять списки внутри JIRA description

Внутри JIRA description использовать Atlassian/Jira markup для списков:

* для списка первого уровня использовать `*`;
* для вложенного списка использовать `**`;
* не использовать Markdown-списки через `-` внутри JIRA panels;
* не смешивать `*` и `-` в одном описании задачи.

Пример:

```
{panel:title=Что нужно сделать?|borderColor=#cbd5e1}
Добавить CapabilityRegistry:

* хранить найденные capabilities;
* хранить найденные skills;
* связывать skill с capability через implements_capability.

Добавить FileSystemCapabilityRegistryLoader:

* читать capability.yaml из capabilities/**/capability.yaml;
* читать SKILL.md из skills/**/SKILL.md.
{panel}
```

## 11. Правила для ссылок и артефактов

Если есть внешние материалы, их желательно добавлять прямо в описание:

- wiki;
- Swagger;
- GitLab repo;
- RFC;
- ADR;
- PRD;
- Vault;
- DevPlatform;
- Time;
- Langfuse;
- RAG docs;
- Grafana dashboard;
- Prometheus query;
- Finedog alert;
- примеры из других команд.

Это особенно важно для задач, где инженер должен опираться на уже существующие решения.

Правила:

- не добавлять ссылки без назначения;
- рядом со ссылкой кратко указать, зачем она нужна;
- не заменять описанием одну ссылку без контекста;
- если ссылка критична для работы, добавить ее в `Что нужно сделать?` или `Зависимости`.

## 12. Правила для observability / metrics / logs задач

Если задача про метрики, логи, алерты, observability:

- писать, зачем это нужно: observability, SLI, Grafana, Finedog, Langfuse, debugging;
- явно указывать low-cardinality / high-cardinality риски, если это важно;
- указывать, где использовать данные: Grafana, Prometheus, Langfuse, Statist, TiMe, логи;
- в критериях успеха фиксировать, что метрика, лог или алерт действительно видны в целевой системе;
- указывать labels, если они уже известны;
- не придумывать labels, если они не были согласованы;
- явно писать, если `thread_id`, `request_id`, `user_id` или другие значения нельзя добавлять в metrics labels из-за high-cardinality.

Хорошие критерии успеха:

- `Метрика agent_tool_calls_total доступна в Prometheus`.
- `В Grafana доступен dashboard с RED-метриками вызовов MCP tools`.
- `Labels не содержат thread_id и других high-cardinality значений`.
- `При ошибке tool call в логах есть error_class и request_id`.

## 13. Правила для agent / AI задач

Если задача касается агента, prompt-ов, HITL, tools, MCPHub, SOP, Rules или capability:

- отделять слои ответственности:
  - Rules;
  - SOP;
  - Knowledge Base;
  - prompt;
  - tool selection;
  - HITL / clarification;
  - observability;
  - runtime;
  - API;
  - frontend;
  - evaluation / benchmark.
- не смешивать всё в одну задачу без необходимости;
- если задача только про backend, это нужно явно писать;
- если frontend будет позже, это нужно фиксировать в контексте или `Что нужно сделать?`;
- если меняется prompt strategy, нужно описать ожидаемое поведение модели;
- если меняется tool selection, нужно указать, какие tools входят в scope;
- если меняется HITL, нужно указать, когда агент должен задавать вопрос пользователю;
- если меняется capability / skill-контент, нужно указать, какие артефакты меняются.

### 13.1. Когда не смешивать задачи

Не стоит смешивать в одну задачу:

- Rules и SOP;
- Knowledge Base и prompt;
- backend runtime и frontend UI;
- tool selection и observability;
- RFC и production implementation;
- capability format и lifecycle process;
- benchmark pipeline и улучшение конкретного prompt.

Если связь есть, но работы независимы, лучше сделать Epic и несколько задач.

## 14. Правила для scope

### 14.1. Если scope понятен

Если scope понятен, нужно явно описать границы работы в `Что нужно сделать?`.

Пример:

```text
- Добавить обработку SendTimeoutError в SSE bridge.
- Проверить, что background producer task корректно завершается.
- Не менять контракт SSE events.
```

### 14.2. Если scope спорный

Если есть риск, что задачу расширят, нужно добавить `Не входит в scope`.

Пример:

```text
{panel:title=Не входит в scope|borderColor=#cbd5e1}
- Изменение публичного SSE-контракта.
- Frontend-обработка новых events.
- Переработка всей observability-модели агента.
{panel}
```

### 14.3. Если задача слишком большая

Если задача содержит несколько независимых направлений, нужно предложить разбиение.

Формат:

```text
Тип задачи: требуется декомпозиция

Рекомендуемое разбиение:
1. Epic: ...
2. Task: ...
3. Task: ...
4. Task: ...
```

## 15. Что должно быть единообразным во всех задачах

Во всех задачах желательно придерживаться одинакового стиля:

- одинаковые названия блоков;
- одинаковый порядок блоков;
- одинаковая глубина детализации;
- одинаковый стиль заголовков;
- одинаковый подход к критериям успеха;
- одинаковый подход к тестированию;
- одинаковый подход к ссылкам;
- одинаковый подход к фиксации неизвестных.
- одинаковый стиль списков внутри JIRA description: `*` для первого уровня, `**` для вложенного уровня;

## 16. Короткий чек-лист перед сохранением задачи

Перед тем как считать задачу хорошо оформленной, нужно проверить:

- Заголовок короткий и понятный?
- Заголовок описывает результат?
- Из контекста ясно, зачем задача нужна?
- Понятно, что именно нужно сделать?
- Границы работы понятны?
- Критерии успеха можно проверить?
- Тестирование описано без воды?
- Есть ссылки на важные материалы?
- Не добавлены факты, которых нет во входных данных?
- Не смешаны ли разные уровни абстракции?
- Если задача исследовательская — есть ли акцент на RFC / анализ вариантов?
- Если задача backend-only — это явно написано?
- Если нужен rollout — он зафиксирован?
- Если есть зависимости — они указаны?
- Если есть неизвестные — они вынесены отдельно?
- Если задача слишком большая — предложена декомпозиция?

## 17. Рекомендуемый стиль работы с ChatGPT при оформлении задач

Если задача формируется через ChatGPT, полезно сразу давать:

- тип задачи: Task / Bug / Epic;
- краткий контекст;
- что уже известно;
- что должно получиться на выходе;
- ссылки на существующие сервисы, API, wiki, репозитории;
- ограничения, например:
  - только backend;
  - frontend позже;
  - API не менять;
  - нужен RFC;
  - rollout поэтапный;
  - нужен только MVP;
  - production-grade enforcement не входит в scope;
- известные зависимости;
- что точно не входит в scope.

Это позволяет получать задачи в более точном и консистентном формате.

### 17.1. Формат ответа ChatGPT при подготовке JIRA-задачи

При генерации задачи возвращать результат в формате:

```text
1. Тип задачи: обычная задача / bug / Epic / требуется уточнение / требуется декомпозиция.
2. Заголовок.
3. Готовое описание для JIRA.
4. Что нужно уточнить — только если критичных данных не хватает.
```

Если данных не хватает для достоверной постановки, не формировать финальное описание задачи.

Если данных хватает для минимальной рабочей постановки, сформировать задачу и вынести неопределенности в `Неизвестно`.

### 17.2. Поведение ChatGPT при сыром контексте

Если пользователь прислал сырой контекст, обсуждение, черновик или заметки, нужно:

1. Определить тип задачи.
2. Проверить, хватает ли данных.
3. Не додумывать факты.
4. Задать только критичные вопросы, если без них задача получится недостоверной.
5. Если данных хватает — переработать входные данные в JIRA-описание.
6. Не переносить обсуждение как есть.
7. Не добавлять опциональные блоки ради формы.
8. Сохранять технические термины на английском, если они используются в проекте.

## 19. Короткая формулировка для Sources

Если нужен максимально короткий meta-guideline для ChatGPT Sources, можно использовать такую формулу:

> Оформляй задачи в JIRA на русском языке. Используй короткие и понятные заголовки, ориентированные на результат. Для обычных задач используй блоки: Контекст, Что нужно сделать?, Критерии успеха, Тестирование. Для багов используй: Проблема, Шаги воспроизведения, Ожидаемый результат, Актуальный результат, Дополнительная информация. Для Epic используй отдельный Epic Name и блоки: Описание, Цель, Обоснование, Преимущества и польза, Стейкхолдеры, Основные аспекты / Функциональные требования, Риски и проблемы, Критерии успеха. Пиши коротко, деловым информационным стилем. Не додумывай факты. Если не хватает критичных данных, задай только вопросы, которые влияют на постановку задачи. Если данных хватает для минимальной рабочей постановки, сформируй задачу и вынеси неопределенности в `Неизвестно`. В описаниях делай акцент на реальном контексте, ожидаемом результате, проверяемости и ссылках на документацию. Не смешивай Rules, SOP, Knowledge Base, prompt-логику, runtime, observability и product context без явной необходимости.
