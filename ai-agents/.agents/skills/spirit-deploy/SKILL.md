# Spirit Deploy CLI

Помогает работать со **Spirit Deploy** (DaaS) через CLI: `dp deploy <command>` (плагин `spirit-deploy` для `dp`).

Две главные задачи:

1. **Готовить и валидировать `application.yaml`** — корректно заполнять placeholder-значения, проставлять `isModified`, согласовывать `environmentType` с `--env`, выбирать кластеры, настраивать пробы/порты/ресурсы.
2. **Выстраивать правильную последовательность команд** — первый деплой (create → do → watch → status) и упрощённый день-2 цикл (обновить образ и задеплоить).

## Когда использовать

Триггеры: пользователь упоминает Spirit Deploy / DaaS, команды `dp deploy …`, файл `application.yaml`, тенант, ревизию, кластер, `serviceId`, либо просит задеплоить, обновить образ, откатить, проверить статус, дождаться завершения деплоя, разобраться, почему не применилась переменная окружения или почему деплой упал.

## Prerequisites: проверка окружения

Перед любыми действиями убедись, что CLI и плагин доступны. Проверка read-only и безопасна.

```bash
# CLI установлен
dp --version

# Плагин spirit-deploy подключён (выводит справку подкоманд deploy)
dp deploy --help
```

Если `dp` не найден или `dp deploy` не распознан — **остановись**. Не готовь `application.yaml` и не выдумывай команды. Сообщи пользователю, что нужно установить `dp` и подключить плагин `spirit-deploy`, и дождись, пока окружение будет готово.

## Правило безопасности (advisory)

Деплой — побочное действие, которое затрагивает рабочие окружения. По умолчанию skill **готовит**, но **не выполняет** изменяющие команды.

- **Не запускай автоматически** изменяющие команды: `create`, `do`, `update`, `rollback`, `shutdown`, `archive`, `restore`, `clone`, `migrate`. Покажи готовую команду целиком и дождись явного подтверждения пользователя.
- **Можно выполнять** read-only команды, когда CLI доступен и нужно свериться с реальным состоянием: `get`, `status`, `watch`.
- **Окружения `production` и `production-payment`** — повышенная осторожность: всегда показывай команду и проси подтверждение, даже если пользователь торопит.
- Всегда отдавай **точные команды и заполненный `application.yaml`**, чтобы пользователь мог запустить их сам. Не придумывай значения `serviceId`, `imageStream`, `tag`, кластеров и переменных — если их нет, помечай как требующие уточнения.

## Рабочий процесс: первый деплой

Полная инструкция «от нуля до running». Перед `application.yaml` нужны: установленный `dp` с плагином `spirit-deploy`, доступ к тенанту, **сервис, зарегистрированный в Software Catalog** (даёт `serviceId` — UUID), и **собранный образ в registry**.

```bash
# 1. Сгенерировать шаблон application.yaml в текущей директории
dp deploy create --new daas --tenant <tenant> --env <env>

# 2. (Вручную) Заполнить application.yaml — см. чеклист ниже

# 3. Зарегистрировать приложение (деплой ещё НЕ выполняется, статус new)
dp deploy create --tenant <tenant> --env <env> --file application.yaml

# 4. Запустить деплой (создаёт ревизию)
dp deploy do --tenant <tenant> --app <app> --env <env> --file application.yaml

# 5. Дождаться результата (опрос каждые 5 с, до 10 мин; exit 0 = running)
dp deploy watch --tenant <tenant> --app <app> --env <env>

# 6. Проверить статус (JSON)
dp deploy status --tenant <tenant> --app <app> --env <env>
```

Обрати внимание: `dp deploy create` делает **две разные вещи** в зависимости от флагов:

- `--new daas` → только **генерирует** локальный шаблон `application.yaml`, ничего не создаёт на сервере.
- `--file <path>` (без `--new`) → **регистрирует** приложение на сервере из заполненного файла.

## Рабочий процесс: день-2 (обновить образ и задеплоить)

После первого деплоя цикл короче. В API-режиме (без `--file`/`--local`) меняется только образ контейнера.

```bash
# Обновить тег/SHA и задеплоить (конфигурация берётся с сервера)
dp deploy do --tenant <tenant> --app <app> --env <env> --tag <tag> --sha <sha>
# Дождаться результата
dp deploy watch --tenant <tenant> --app <app> --env <env>
```

Если нужно изменить и **конфигурацию** (ресурсы, реплики, переменные, пробы), а не только образ — деплой полным файлом: `dp deploy do … --file application.yaml` (или `--local`). См. [режимы](#режимы-file--local--api).

## Ключевые концепции и типичные ошибки

Большинство проблем со Spirit Deploy — из этого списка. Проверяй его при разборе ошибок.

### service.yaml ≠ application.yaml

Это **два разных файла для двух разных систем**, и они **не связаны автоматически**:

- `service.yaml` — описывает **сервис** в Software Catalog (что это за сервис). Даёт `id` сервиса в формате UUID.
- `application.yaml` — описывает **как сервис задеплоен** в Spirit Deploy (кластеры, ресурсы, реплики, стратегия).

Поля из `service.yaml` **не переносятся** в `application.yaml`. Связь — только через `serviceId` (UUID сервиса). Список переменных в `envs` приложения должен совпадать со списком `envs` из `service.yaml`.

### environmentType должен совпадать с --env

Флаг `--env` **обязателен** для всех команд. Допустимые значения: `development`, `production`, `production-payment`.

Поле `environmentType` внутри `application.yaml` **должно точно совпадать** со значением `--env`. Несовпадение — частая причина ошибки. Чтобы задеплоить в dev и prod, запусти команду дважды — с разными `--env` и соответствующими файлами.

### isModified: true — обязателен при изменении переменной

Это ошибка №1. Поле `isModified` встречается на нескольких уровнях:

- На уровне **приложения и компонента** — read-only, приходит с сервера.
- На уровне **переменной окружения** — **управляешь ты**. Это сигнал серверу «значение изменилось, перезапиши его».

**Без `isModified: true` сервер проигнорирует новое значение переменной.** Поэтому при добавлении или изменении любой переменной всегда ставь `isModified: true`.

### serviceId — это UUID из Software Catalog

`serviceId` в контейнере — UUID сервиса, зарегистрированного в Software Catalog (`Код` → `Каталог ПО`), а не имя сервиса и не значение из `service.yaml`. Если UUID неизвестен — пометь как требующий уточнения, не выдумывай.

### Переменные окружения — только plain-text

`envs` принимают **только строковые значения**. Интеграция с Vault и ссылки на секреты не поддерживаются. Пустые значения недопустимы.

### Кластеры и placement зависят от --env

Кластеры задаются в секции `placement.clusters`. Доступный набор кластеров зависит от окружения (`--env`). Не подставляй кластер из другого окружения.

```yaml
placement:
    type: k8s          # k8s | vm | stub
    clusters:
        - <cluster>
```

### Режимы: --file / --local / API

Команды `update` и `do` берут конфигурацию одним из трёх способов:

| Режим | Как включить | Что происходит |
|-------|--------------|----------------|
| Файл по пути | `--file ./path/app.yaml` | Полное обновление конфигурации из указанного файла |
| Локальный файл | `--local` | Полное обновление из `application.yaml`/`application.json` в текущей директории |
| API (по умолчанию) | без флагов | Минимальное обновление: только образ (`--tag`/`--sha`/`--branch`) |

Выбор режима важен: API-режим **не** применит изменения ресурсов/переменных/проб — для них нужен `--file` или `--local`.

### Имя файла для --local — application.yaml

При `--local` CLI ищет именно `application.yaml` или `application.json` в текущей директории. Имя `app.yaml` **не поддерживается**. Для произвольного имени используй `--file ./my-name.yaml`.

## Заполнение application.yaml

Сгенерируй шаблон (`dp deploy create --new daas …`) или возьми его из [references/application-yaml.md](references/application-yaml.md), затем замени placeholder-значения:

| Placeholder | Что подставить | Где взять |
|-------------|----------------|-----------|
| `<application-name>` | Имя приложения в Spirit Deploy | Поле `name` из `service.yaml` (Software Catalog) |
| `<component-name>` | Имя компонента (Deployment в K8s) | На своё усмотрение |
| `<service-id>` | UUID сервиса | Software Catalog |
| `<image-stream>` | Имя image stream (без тега) | Из пайплайна |
| `<tag>` | Тег образа | Release tag из пайплайна |
| `<sage-group>` | Группа Sage для логирования | Конфигурация команды (можно пусто) |
| `<envs>` | Переменные окружения | Вручную, только строки; список — как в `service.yaml` |

Критические правила при заполнении:

- `environmentType` = значению `--env`.
- Каждой новой/изменённой переменной — `isModified: true`.
- `clusters` — из набора, доступного для выбранного `--env`.
- `resources` в формате Kubernetes: `cpu: 250m`, `mem: 256Mi`.
- Пробы (`livenessProbe`/`readinessProbe`/`startupProbe`) — под реальный сервис (путь, порт, таймауты).
- `serviceId`, `imageStream`, `tag` — реальные значения; неизвестные помечай явно.

Полный справочник всех полей, типов, обязательности и примеры (пробы http/grpc/tcp, `trafficManagement`, параметры для `deployment`/`statefulset`/`crontask`/`task`) — в [references/application-yaml.md](references/application-yaml.md).

## Команды CLI

| Команда | Назначение |
|---------|------------|
| `dp deploy create --new daas` | Сгенерировать локальный шаблон `application.yaml` |
| `dp deploy create --file …` | Создать (зарегистрировать) приложение |
| `dp deploy do` | Запустить деплой (создаёт ревизию) |
| `dp deploy update` | Обновить конфигурацию без деплоя |
| `dp deploy get` | Получить текущую конфигурацию |
| `dp deploy status` | Статус приложения (JSON) |
| `dp deploy watch` | Дождаться завершения деплоя |
| `dp deploy rollback` | Откатить на предыдущую ревизию |
| `dp deploy shutdown` | Остановить ревизию |
| `dp deploy archive` / `restore` | Архивировать / восстановить приложение |
| `dp deploy clone` | Клонировать приложение |
| `dp deploy migrate` | Миграция из K8s в Spirit Deploy (без токена) |

Детали команд, флагов, режимов, поведение `watch` (коды возврата) и сценарии — в [references/commands.md](references/commands.md). Авторитетный список флагов и переменных окружения конкретной команды — `dp deploy <command> --help`; не выдумывай флаги и env-переменные, которых нет в справочнике.

## Аутентификация и установка

- Установка плагина: `dp plugin install --name spirit-deploy`.
- Все команды, кроме `migrate`, требуют авторизационный токен.
