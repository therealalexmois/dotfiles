## Установка и аутентификация

```bash
dp plugin install --name spirit-deploy
```

Все команды, кроме `migrate`, работают с авторизационным токеном.

## Общие флаги

Встречаются в большинстве команд (см. `--help` по каждой команде для полного списка):

| Флаг | Назначение |
|------|------------|
| `--tenant <tenant>` | Тенант в Spirit Deploy |
| `--env <env>` | Окружение: `development`, `production`, `production-payment`. Обязателен |
| `--app <app>` | Имя приложения |
| `--file <path>` | Путь к файлу конфигурации (`update`, `do`, `create`) |
| `--local` | Читать `application.yaml`/`application.json` из текущей директории |
| `--new daas` | Для `create`: сгенерировать локальный шаблон вместо создания приложения |
| `--tag <tag>` | Тег образа (день-2 деплой) |
| `--sha <sha>` | SHA коммита |
| `--branch <branch>` | Имя ветки |

`--env` обязателен для всех команд, и его значение должно совпадать с `environmentType` в `application.yaml`.

## Режимы конфигурации: --file / --local / API

Команды `update` и `do` получают конфигурацию одним из трёх способов:

| Режим | Как включить | Что происходит |
|-------|--------------|----------------|
| Файл по пути | `--file ./path/app.yaml` | Полное обновление конфигурации из указанного файла |
| Локальный файл | `--local` | Полное обновление из `application.yaml`/`application.json` в текущей директории |
| API (по умолчанию) | без флагов | Минимальное обновление: только образ (`--tag`/`--sha`/`--branch`) |

- API-режим меняет только образ контейнера; ресурсы, реплики, переменные и пробы **не** трогаются.
- `--file` и `--local` применяют полную конфигурацию из локального файла.
- При `--local` имя файла строго `application.yaml` или `application.json` (не `app.yaml`).

## Команды

### create — создать приложение / сгенерировать шаблон

Две роли в зависимости от флагов:

```bash
# Сгенерировать локальный шаблон application.yaml (ничего не создаётся на сервере)
dp deploy create --new daas --tenant <tenant> --env <env>

# Зарегистрировать приложение из заполненного файла (статус new, деплой не запускается)
dp deploy create --tenant <tenant> --env <env> --file application.yaml
```

Пример вывода при создании:

```
10:55:51 INF creating new application in tenant "invest-core"
10:55:52 INF application "spirit-cli-sandbox" created
```

### do — запустить деплой

Создаёт ревизию и запускает деплой; в выводе появляется ссылка на страницу деплоя в UI.

```bash
# Полный деплой из файла (первый деплой или изменение конфигурации)
dp deploy do --tenant <tenant> --app <app> --env <env> --file application.yaml

# День-2: обновить только образ (API-режим)
dp deploy do --tenant <tenant> --app <app> --env <env> --tag <tag> --sha <sha>
```

### update — обновить конфигурацию без деплоя

Применяет конфигурацию (поддерживает режимы `--file`/`--local`/API), но не запускает деплой.

### get — получить текущую конфигурацию

Возвращает конфигурацию приложения с сервера. На уровне приложения и компонента поле `isModified` read-only.

### status — статус приложения

```bash
dp deploy status --tenant <tenant> --app <app> --env <env>
```

Возвращает JSON с текущим состоянием приложения и активной ревизии.

### watch — дождаться завершения деплоя

```bash
dp deploy watch --tenant <tenant> --app <app> --env <env>
```

Опрашивает статус каждые 5 секунд в течение 10 минут. См. [коды возврата](#поведение-watch-и-коды-возврата).

### rollback — откат на предыдущую ревизию

### shutdown — остановка ревизии

### archive / restore — архивировать / восстановить приложение

### clone — клонировать приложение

### migrate — миграция из K8s в Spirit Deploy

Единственная команда, не требующая авторизационного токена.

## Поведение watch и коды возврата

`dp deploy watch` опрашивает статус каждые 5 секунд до 10 минут:

- Код возврата `0` — приложение достигло статуса `running`.
- Ненулевой код — `failed`, `archived` или таймаут.

Это позволяет использовать `watch` в скриптах как шаг ожидания результата деплоя.

## Типичные сценарии

Первый деплой (от нуля до running):

```bash
dp deploy create --new daas --tenant <tenant> --env <env>     # 1. шаблон
# 2. заполнить application.yaml
dp deploy create --tenant <tenant> --env <env> --file application.yaml   # 3. создать
dp deploy do --tenant <tenant> --app <app> --env <env> --file application.yaml  # 4. деплой
dp deploy watch --tenant <tenant> --app <app> --env <env>     # 5. ждать
dp deploy status --tenant <tenant> --app <app> --env <env>    # 6. статус
```

День-2 (обновить образ и задеплоить):

```bash
dp deploy do --tenant <tenant> --app <app> --env <env> --tag <tag> --sha <sha>
dp deploy watch --tenant <tenant> --app <app> --env <env>
```

Деплой в dev и prod — запусти команды дважды, с разными `--env` и соответствующими файлами `application.yaml` (где `environmentType` совпадает с `--env`).

## Переменные окружения CLI

Вместо флагов CLI поддерживает переменные окружения. Полный список — в `dp deploy --help` /
на странице commands.md документации. Конкретные имена переменных здесь не приводятся,
чтобы не выдумывать их; уточняй через `--help`.
