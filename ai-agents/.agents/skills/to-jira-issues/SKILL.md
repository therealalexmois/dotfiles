# To JIRA Issues

Декомпозиция входного документа (PRD, RFC, ADR, Epic, контекст инициативы) в набор готовых JIRA-задач для инженерной реализации в проектах Data Agent / Data Platform.

Цель — получить проверяемые vertical slices, оформленные в едином стиле, без додумывания фактов и без смешивания уровней абстракции.

## Source of truth (обязательно)

**Перед формированием любых задач прочитай `references/jira_issue_authoring.md` и следуй ему как обязательному контракту оформления.** Это основной источник истины для:

* языка и стиля (русский, деловой информационный стиль; общие правила русской технической редактуры — skill `russian-technical-editor`);
* структуры блоков для Task, Bug, Epic;
* JIRA-разметки (`{panel:...}`, списки через `*` и `**`);
* правил «не додумывать факты» и работы с блоком `Неизвестно`;
* формата ответа `требуется уточнение` / `требуется декомпозиция`;
* чек-листа перед сдачей задачи.

Не создавай собственный generic issue template поверх `references/jira_issue_authoring.md`. Если этот skill и `jira_issue_authoring.md` расходятся в деталях оформления — приоритет у `jira_issue_authoring.md`.

## Когда активируется

Используй skill, когда на входе есть документ или контекст уровня инициативы (PRD, RFC, ADR, Epic description, дизайн-заметки, обсуждение), и нужно превратить его в задачи на реализацию. Типичные формулировки: «разбей это на JIRA-задачи», «декомпозируй эпик», «сделай тикеты на реализацию», «разнеси по vertical slices», «подготовь задачи для команды».

Не используй skill для оформления одной уже сформулированной задачи без декомпозиции — для этого достаточно напрямую следовать `references/jira_issue_authoring.md`.

## Обязательные правила поведения

1. Не публикуй задачи в issue tracker автоматически.
2. Не создавай и не изменяй Jira issues без явной команды пользователя.
3. По умолчанию возвращай draft задач в чате.
4. Не додумывай отсутствующие факты. Если данных недостаточно:
   * при **критическом** пробеле верни только вопросы, необходимые для достоверной декомпозиции (см. «Поведение при неполных данных»);
   * при **некритическом** пробеле сформируй задачу и вынеси неопределённость в блок `Неизвестно`.
5. Не добавляй rollout, metrics, feature flags, endpoints, owners, labels, dependencies, если они не указаны во входных данных.
6. Сохраняй технические термины, названия компонентов, API, paths и identifiers as-is.
7. Не смешивай product scope, architecture decision и implementation tasks в одной задаче.

## Workflow декомпозиции

### 1. Определи тип входа и scope

Определи, что на входе: PRD (продукт), RFC/ADR (архитектурное решение), Epic (инициатива) или сырой контекст. От этого зависит уровень декомпозиции и допустимая глубина implementation details.

* Для RFC / ADR / архитектурного решения **не притягивай implementation details, пока они не зафиксированы исходным документом.** Если способ реализации ещё не выбран — задача должна быть исследовательской (RFC / анализ вариантов), а не реализацией кода (см. раздел 8 в `references/jira_issue_authoring.md`).
* Для PRD / Epic выдели продуктовый и инженерный слои, не смешивая их в одной задаче.

### 2. Проверь полноту данных

Проверь, хватает ли данных для достоверной декомпозиции. Не додумывай факты. Реши по каждому пробелу, критический он или некритический (см. «Поведение при неполных данных»).

### 3. Сформируй vertical slices

Разбей работу на vertical slices по правилам ниже. Определи зависимости между ними (dependency ordering: блокеры первыми).

### 4. Классифицируй каждую задачу

Для каждого slice определи тип задачи (`Task` / `Bug` / `Epic` / `требуется уточнение`) и execution mode (`AFK` / `HITL`).

### 5. Верни результат

Верни сводку, затем задачи в формате из «Output contract». Каждая задача оформляется строго по `references/jira_issue_authoring.md`.

## Правила vertical slices

Все implementation tasks формируются как vertical slices:

* каждая задача должна давать **самостоятельный проверяемый результат** — её можно проверить через поведение системы, тест, лог, метрику, UI, API-ответ или документ;
* не создавай задачи, которые являются только техническим слоем без проверяемого результата, **если это не обосновано архитектурным ограничением** (например, требуется отдельный RFC до реализации);
* предпочитай несколько тонких slices одному толстому;
* у каждого slice — проверяемые acceptance criteria (блок `Критерии успеха` по контракту).

Для исследовательской / RFC / ADR-задачи проверяемым результатом является RFC, анализ вариантов или согласованное решение, а не код.

## Классификация HITL / AFK

Для каждой задачи добавляй execution mode с одной короткой причиной:

* **AFK** — задачу можно реализовать по зафиксированным требованиям без дополнительного решения человека.
* **HITL** — до завершения задачи требуется решение, согласование, review или выбор человека (например, архитектурный выбор, дизайн-review, согласование с другой командой, выбор между вариантами).

`HITL / AFK` относится к **выполнению инженерной задачи**, а не к runtime HITL-механизму Data Agent. Если задача описывает реализацию runtime-HITL агента, это содержание задачи, а не её execution mode.

Предпочитай AFK, где это возможно. Если задача требует решения, которое ещё не зафиксировано в исходном документе — это HITL.

## Поведение при неполных данных

**Критический пробел** — данные, без которых задача получится недостоверной. Влияет на: тип задачи, заголовок, scope, ожидаемое поведение, критерии успеха, способ проверки, архитектурные ограничения, зависимости. При критическом пробеле **не формируй финальное описание задачи** — верни только блок с вопросами:

```text
Тип задачи: требуется уточнение

Что нужно уточнить:
1. ...
2. ...
```

Вопросы должны быть короткими, конкретными, связанными с результатом, не больше необходимого минимума.

**Некритический пробел** — неопределённость, которую можно явно зафиксировать, не блокируя минимальную рабочую постановку. В этом случае сформируй задачу и вынеси неопределённость в блок `Неизвестно` (формат — в `references/jira_issue_authoring.md`).

Не задавай вопросы ради процесса, если данных хватает для минимальной постановки или вопрос касается реализации, которую выбирает исполнитель.

## Output contract

Сначала верни краткую **сводку**:

* исходный документ / инициатива;
* выбранный уровень декомпозиции;
* список vertical slices;
* зависимости между задачами;
* риски или неизвестные, влияющие на декомпозицию.

Затем для **каждой задачи** верни:

1. **Тип задачи**: `Task`, `Bug`, `Epic` или `требуется уточнение`.
2. **Execution mode**: `AFK` или `HITL`, с одной короткой причиной.
3. **Заголовок**.
4. **Зависимости** от других задач, если есть.
5. **Готовое описание для JIRA** строго по `references/jira_issue_authoring.md` (соответствующий шаблон Task / Bug / Epic, JIRA-разметка, списки через `*` и `**`, опциональные блоки только при реальной необходимости).

Если для конкретного slice критически не хватает данных — вместо описания верни блок `Что нужно уточнить` для этого slice.

Если входная инициатива слишком большая и содержит несколько независимых направлений — предложи разбиение в формате `требуется декомпозиция` (см. `references/jira_issue_authoring.md`, раздел 14.3).

## Ограничения

* Язык результата — **русский**. Технические термины, названия компонентов, API, paths и identifiers — as-is.
* Не публикуй и не изменяй задачи в трекере автоматически.
* Не смешивай product scope, architecture decision и implementation tasks в одной задаче.
* Не добавляй rollout, metrics, feature flags, endpoints, owners, labels или dependencies, если они не указаны во входных данных.
* Не добавляй собственный generic issue template поверх `references/jira_issue_authoring.md`.
* Для RFC / ADR / архитектурных задач не притягивай implementation details, пока они не зафиксированы исходным документом.
* Не додумывай факты — используй `Неизвестно` и уточняющие вопросы по правилам выше.
