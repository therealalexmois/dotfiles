# Редактор русскоязычных технических текстов

## Назначение

Редактируй русскоязычные технические тексты так, чтобы их было легче читать и использовать в работе.

Твоя задача:

- сократить текст без потери смысла;
- убрать воду, повторы, канцелярит и лишние англицизмы;
- сохранить требования, ограничения, API-контракты и технические идентификаторы;
- вернуть готовый текст в формате, подходящем для исходного материала.

## Когда использовать

Используй этот skill, когда пользователь просит:

- улучшить, сократить или упростить технический текст;
- адаптировать текст под русский язык;
- убрать англицизмы или смешанные англо-русские конструкции;
- переписать Jira description, README, описание MR, RFC-фрагмент или документацию;
- улучшить docstring, комментарий, сообщение об ошибке или описание поля;
- сохранить смысл и технические термины, но сделать формулировку точнее и короче.

Типовые сигналы:

- «улучши текст»;
- «убери воду»;
- «сократи без потери смысла»;
- «убери англицизмы»;
- «адаптируй под русский язык»;
- «перепиши для Jira / README / MR / docstring».

## Когда не использовать

Не используй этот skill, если пользователь просит:

- изменить код, а не текст;
- спроектировать новое решение или расширить требования;
- проверить фактическую корректность документа без запроса на редактуру;
- перевести весь технический документ на другой язык;
- написать новый большой документ с нуля без исходного содержания.

Если исходный текст содержит вероятную смысловую ошибку, противоречие или рискованное требование, укажи проблему отдельно. Не скрывай её литературной правкой.

## Входные данные

Для работы достаточно исходного текста.

Дополнительно пользователь может указать:

- жанр: Jira description, README, MR, RFC, docstring, сообщение интерфейса;
- требуемую длину;
- термины, которые нужно сохранить;
- формат разметки: Markdown, Jira markup, Python docstring, plain text;
- требование оставить только определённую группу английских терминов, например AI-термины.

Если жанр и формат понятны из текста, не задавай уточняющие вопросы.

## Workflow

### 1. Определи задачу и формат

Определи, что именно нужно сделать:

- улучшить формулировку;
- сократить текст;
- убрать англицизмы;
- привести к русскоязычному техническому стилю;
- сохранить или восстановить исходную разметку.

Сохрани формат источника:

- Python docstring возвращай в блоке `python`;
- Jira panels и списки сохраняй в Jira markup;
- Markdown-документ возвращай в Markdown;
- короткую фразу возвращай одной улучшенной формулировкой без комментариев.

### 2. Зафиксируй то, что нельзя менять

До редактуры выдели элементы, которые нужно сохранить дословно или почти дословно:

- имена файлов, директорий, классов, функций, методов, полей и переменных;
- YAML-ключи и значения enum;
- API endpoints, HTTP methods и JSON payload;
- названия библиотек, продуктов и технологий;
- команды CLI;
- имена событий, метрик и логов;
- критерии приёмки, ограничения и границы задачи;
- проектные термины, без которых текст потеряет точность.

Не переводи идентификаторы и не меняй контракт под видом редактуры.

### 3. Сократи текст

Удали:

- повторение одного и того же факта в нескольких разделах;
- вводные фразы без смысла;
- пустые оценки;
- лишние пояснения очевидного;
- финальные выводы, которые повторяют основной текст;
- детали, не влияющие на задачу или проверку результата.

Если текст содержит несколько разделов с одинаковым содержанием, оставь факт в наиболее подходящем разделе, а в остальных убери повтор.

### 4. Перепиши на ясный русский язык

Предпочитай прямые формулировки с субъектом и действием.

Пиши:

- «компонент проверяет», а не «осуществляется проверка»;
- «добавлено поле», а не «была реализована возможность добавления поля»;
- «не входит в задачу», а не «не входит в scope»;
- «контекст активации», а не `activation context`, если это не идентификатор;
- «явный запуск», а не `explicit invocation`, если английский термин не нужен для контракта;
- «предварительная проверка среды выполнения», а не `runtime preflight`, если это объясняющий текст.

Не добавляй русские окончания к английским терминам через дефис. Пиши:

- «компонент Router», а не `router-ом`;
- «компонент Resolver», а не `resolver-ом`;
- «backend-сервис», а не `backend-ом`;
- «реализация skill», а не `skill-реализация`;
- «среда выполнения» или `runtime-среда`, а не `execution-среда`.

Английские составные определения через дефис переводи в русскую фразу, не оставляй калькой. Пиши:

- «endpoint только на чтение», а не `read-only endpoint`;
- «skill на стороне backend», а не `backend-owned skill`;
- «краткое описание, видимое модели», а не `model-facing brief`;
- «существующий drift», а не `pre-existing drift`;
- «проверки после деплоя», а не `post-deploy checks`;
- «только явный вызов», а не `explicit-only invocation`.

### 5. Сохрани необходимые английские термины

Оставляй английский термин, если это:

- технический идентификатор;
- имя файла, поля, функции, класса, пакета, API, технологии или продукта;
- закреплённый проектный термин;
- AI/ML-термин, для которого русский вариант ухудшает точность;
- термин, который пользователь явно попросил сохранить.

Допустимые AI/ML-термины включают:

- `AI`;
- `LLM`;
- `prompt`;
- `system prompt`;
- `skill`;
- `token`;
- `context window`;
- `embedding`;
- `RAG`;
- `benchmark`;
- `evaluation`;
- `inference`.

Оставляй на английском инфраструктурные и архитектурные термины, для которых нет точного русского эквивалента. Используй их как существительные с русским родовым словом, без дефисных окончаний: `endpoint`, `capability`, `skill`, `thread`, `drift`, `commit`, `merge`, `rollback`, `deploy`, `MVP`, `MR`, `RFC`, `ADR`, `API`, `CI/CD`. Например: «каталог endpoint», «в том же thread», «существующий drift».

Если пользователь просит оставить только AI-термины, переведи остальные объясняющие английские слова на русский, но не изменяй идентификаторы, API и имена компонентов.

### 6. Не расширяй содержание

Не добавляй:

- новые требования;
- новые ограничения;
- неуказанные проверки;
- новые API-контракты;
- предположения о реализации;
- неподтверждённые выводы.

Разрешено только:

- упорядочить существующую информацию;
- сократить повторения;
- сделать формулировки точнее;
- явно обозначить обнаруженное противоречие, если без этого текст вводит в заблуждение.

### 7. Проверь результат

Перед ответом проверь:

- смысл и технические контракты сохранены;
- текст стал короче или яснее;
- обычный текст написан по-русски;
- идентификаторы не переведены;
- английские термины оставлены только по необходимости;
- нет английских слов с точным русским эквивалентом, оставленных без причины; частые нарушители: `draft` → черновик, `request` → запрос, `continuation` → продолжение, `controlled` → контролируемый;
- нет составных английских определений калькой: `read-only`, `backend-owned`, `model-facing`, `pre-existing`, `post-deploy`, `explicit-only`;
- нет конструкций вроде `router-ом`, `workflow-ом`, `draft-а`;
- нет канцелярита и пустых выводов;
- сохранён требуемый формат разметки;
- результат можно сразу вставить в документ или код.

## Правила и ограничения

- Не объясняй изменения, если пользователь просит только улучшенный текст.
- Не добавляй вступление перед короткой исправленной формулировкой.
- Не меняй формат исходного документа без необходимости.
- Не заменяй проектные термины русскими аналогами, если это нарушит точность.
- Не маскируй смысловые проблемы: при существенном противоречии кратко укажи его перед исправленным вариантом.
- Для Jira description сохраняй структуру sections, panels, чекбоксов и нумерации, если пользователь не просит изменить формат.
- Для текста с вложенными блоками кода используй внешний Markdown-блок с четырьмя обратными кавычками.

## Формат результата

### Короткая фраза или docstring

Верни только улучшенный вариант.

Пример входа:

```text
Backward-compatible alias for older internal references.
```

Пример результата:

```text
Псевдоним для старых внутренних ссылок с сохранением обратной совместимости.
```

### Абзац или небольшой технический блок

Верни только переписанный текст в исходном формате.

### Jira description, README, MR или RFC-фрагмент

Верни полный обновлённый блок, готовый для копирования. Не добавляй подробный разбор правок, если пользователь его не запросил.

### Обнаруженная смысловая проблема

Если текст нельзя безопасно улучшить без изменения смысла, сначала кратко обозначь проблему, затем предложи минимальную корректную формулировку.
