# Thread message (тред к Time post)

Операционные детали, слишком длинные или шумные для основного Time post. Общие
правила - [announcement-rules.md](../references/announcement-rules.md).

## Когда создавать

Хотя бы одно условие: есть длинные Sage-ссылки; есть errors/warnings/known
issues; метрики изменились после релиза; основной пост превысил бы лимит Time;
нужна отдельная post-release observation; release manager фиксирует решение.

Не создавать тред только чтобы повторить основной пост.

## Структура

```
**Post-release observation**: T+2-3 min

**<метка наблюдения>**:
- <пункт наблюдения>
- <пункт; Status указывай только если он изменился относительно основного поста>

**Logs**:
- [<подпись>](<кликабельная Sage-ссылка на error/metric, сгруппированная по сигнатуре>)
- [<подпись>](<...>)

**Decision**: <rollback not required / observing / rollback in progress / follow-up required>.
**Follow-up**: <action>. **Owner**: @owner.

Next observation: T+30 min, если нужно.
```

## Правила

Sage - источник observability, не «просто логи». Длинные Sage-ссылки прикладывать
когда релевантны. Повторяющиеся ошибки группировать по сигнатуре. Решение
rollback/follow-up - явно. Для каждого issue/follow-up - owner. Без user-facing
формулировок. Traces не упоминать, если их нет.

- Ссылки в `Logs` делай кликабельными (`[подпись](url)`), не текстом. Для deep-
  ссылки на конкретный тип ошибки точную строку `message=` бери из Sage (запрос
  по `version=` + `level="ERROR"`), внутренние кавычки экранируй `\"`, URL
  собирай через urlencode. Метку блока называй `Logs`, не `Sage links`.
- Не дублируй с основным постом Status, Rollout и changelog - тред несет только
  то, чего в основном посте нет (регрессия, deep-ссылки, decision, follow-up).
- Перечисления выводи ненумерованным списком, а не одним плотным предложением;
  у блока-перечисления жирная метка с двоеточием, буллеты вплотную к ней.

Тайминг: `T+2-3 min` - быстрое наблюдение по критичным post-release проблемам;
`T+30 min` - поздняя проверка стабильности, если нужно.
