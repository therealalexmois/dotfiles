# Writing Rules from Pishi Sokrashchay

## Назначение

Этот source фиксирует основные правила написания и редактирования текста по принципам книги «Пиши, сокращай».

Использовать эти правила для рабочих текстов, заметок, RFC, JIRA-задач, писем, документации, постов, презентаций и промптов.

Главная идея: сильный текст — это не просто короткий текст. Сильный текст решает задачу читателя: быстро дает пользу, ясно объясняет мысль, не заставляет продираться через шум и помогает принять решение.

## 1. Начинать с задачи читателя

Перед написанием текста нужно понять:

- кто будет читать;
- зачем читатель откроет текст;
- что он должен понять, решить или сделать;
- что изменится после чтения.

Плохая стартовая позиция: «Я хочу рассказать про проект».

Хорошая стартовая позиция: «Читателю нужно понять, почему проект задерживается, какие есть риски и какое решение требуется».

Текст должен иметь полезное действие. Если непонятно, какую пользу получает читатель, текст нужно пересобрать или не писать.

## 2. Писать для конкретной аудитории

Нельзя писать один универсальный текст «для всех». Один и тот же объект нужно описывать по-разному для разных читателей.

| Аудитория | Что ей важно |
| --------- | ------------ |
| Клиент | Какую проблему решит продукт |
| Руководитель | Риски, сроки, деньги, решение |
| Инженер | Архитектура, ограничения, зависимости |
| Аналитик | Бизнес-правила, сценарии, исключения |
| Инвестор | Рынок, экономика, рост, устойчивость |

Перед написанием нужно выбрать основного читателя и писать под его задачу.

## 3. Сначала структура, потом слова

Если текст еще не написан, работать сверху вниз:

1. полезное действие;
2. структура;
3. абзацы;
4. предложения;
5. слова;
6. подача.

Если текст уже написан, редактировать можно снизу вверх:

1. убрать лишние слова;
2. упростить предложения;
3. разделить или перестроить абзацы;
4. пересобрать структуру;
5. проверить полезное действие.

Не нужно «красиво переписывать» плохую структуру. Сначала нужно проверить, тот ли материал, в том ли порядке и для того ли читателя.

## 4. Начинать с сути

Введение не должно быть обязательным разогревом.

Для рабочих текстов сильное начало обычно содержит:

- главный вывод;
- проблему;
- решение;
- риск;
- требуемое действие;
- статус.

Плохо начинать с исторической справки, общих слов и длинного контекста без вывода.

Хороший старт:

> Проект задерживается на две недели из-за блокера в интеграции. Нужно выбрать один из двух вариантов: временный workaround или перенос релиза.

## 5. Структура должна помогать задаче текста

Нет универсальной структуры для всех текстов. Структура зависит от задачи.

| Задача | Подходящая структура |
| ------ | -------------------- |
| Объяснить проблему | Суть → причина → последствия → решение |
| Предложить решение | Проблема → варианты → рекомендация → риски |
| Написать отчет | Выводы → факты → детали → приложения |
| Продать продукт | Аудитория → боль → сценарии → свойства → доказательства |
| Сделать инструкцию | Когда применять → шаги → исключения → проверка результата |

Структура должна помогать читателю быстрее понять смысл, а не следовать шаблону ради шаблона.

## 6. Один абзац — одна мысль

Абзац нужен для управления вниманием.

Хороший абзац:

- раскрывает одну мысль;
- начинается с понятной темы;
- не смешивает разные вопросы;
- сохраняет логичный порядок;
- помогает быстро просканировать текст глазами.

Если в абзаце есть несколько мыслей, его нужно разделить.

## 7. Одно предложение — одна основная мысль

Перегруженное предложение заставляет читателя перечитывать.

Признаки перегруза:

- много запятых;
- несколько вложенных уточнений;
- длинные причастные и деепричастные обороты;
- много однородных членов;
- непонятно, кто действует;
- предложение нельзя быстро пересказать.

Что делать:

- выбрать главного героя;
- вынести обстоятельства в отдельные предложения;
- длинные перечисления превратить в список или таблицу;
- сначала дать суть, потом детали.

## 8. Писать о людях, действиях и событиях

Текст становится мутным, когда в нем много абстрактных процессов.

Плохо:

- осуществляется реализация;
- производится обработка;
- наблюдается ухудшение;
- имеет место повышение эффективности.

Лучше:

- команда реализует;
- сервис обрабатывает;
- пользователи чаще получают ошибку;
- новая логика сокращает время обработки.

Читатель должен понимать, кто что делает, что происходит и к чему это приводит.

## 9. Убирать слова, которые не несут смысла

Стоп-слова не запрещены. Они сигнализируют, что слово нужно проверить.

Чаще всего лишними бывают:

- вводные конструкции: `безусловно`, `очевидно`, `как известно`, `на мой взгляд`;
- неопределенность: `некоторые`, `ряд`, `примерно`, `различные`, `в какой-то степени`;
- заумь: слова, которые создают видимость профессиональности, но не уточняют смысл;
- оценки: `качественный`, `эффективный`, `надежный`, `уникальный`;
- штампы: `команда профессионалов`, `индивидуальный подход`, `широкий спектр услуг`.

Удаление лишних слов само по себе не делает текст сильным. После сокращения текст нужно наполнить фактами, примерами и полезной информацией.

## 10. Заменять оценки фактами

Оценки редко убеждают сами по себе. Они показывают вывод автора, но не дают читателю оснований согласиться.

Плохо:

> У нас надежный сервис.

Лучше:

> Сервис работает в двух зонах доступности, имеет SLO 99,9%, алерты по ошибкам и runbook для восстановления после отказа.

Оценку нужно заменять на:

- факт;
- сценарий;
- пример;
- демонстрацию;
- ограничение;
- последствие.

Не нужно писать «важно». Нужно показать, на что это влияет.

## 11. Не прятать плохие новости за канцеляритом

Канцелярит создает видимость аккуратности, но часто прячет смысл.

Плохо:

> В рамках выполнения работ наблюдается отклонение от плановых сроков.

Лучше:

> Работа задерживается на две недели из-за блокера в интеграции.

Плохие новости нужно писать спокойно, прямо и проверяемо: что случилось, на что влияет, что делаем дальше.

## 12. Учитывать уровень подготовки читателя

Читатель понимает текст через то, что уже знает.

Правила:

- новичку нужно объяснить термины;
- эксперту не нужно объяснять базу;
- смешанной аудитории нужны короткие определения и примеры;
- если термин неизбежен, нужно показать его через сценарий или последствие.

Для технических текстов можно использовать термины вроде `SSE`, `middleware`, `backpressure`, `retry`, если аудитория их понимает. Если аудитория смешанная, нужно объяснить, как термин влияет на продукт, эксплуатацию или решение.

## 13. Тон должен соответствовать ситуации

Тональность складывается из выбора слов, дистанции, степени формальности, подробности и контекста площадки.

Правила:

- не изображать дружелюбие, если текст про проблему;
- не прятать плохие новости за канцеляритом;
- не писать слишком свободно там, где нужна точность;
- не использовать корпоративные штампы вместо смысла;
- не путать спокойный тон с бюрократическим языком.

Тон должен помогать задаче текста, а не демонстрировать стиль автора.

## 14. Жанр диктует структуру

Разные жанры требуют разных правил.

| Жанр | Главное правило |
| ---- | --------------- |
| Текст о себе | Писать не о себе, а о пользе для читателя |
| Текст о компании | Отвечать на вопросы клиентов, сотрудников или партнеров |
| Рабочий документ | Быстро дать выводы, факты и материал для решения |
| Пресс-релиз | Начинать с инфоповода |
| Холодное письмо | Писать персонально, коротко, с простым следующим шагом |
| Слайды | Делать капсулы смысла, а не текст для чтения с экрана |
| Промостраница | Одна аудитория, одна задача, один набор сценариев |

Нельзя применять одну структуру ко всем жанрам.

## 15. Короткий текст не всегда сильный

Сокращение — не самоцель.

Короткий, но пустой текст хуже длинного, но полезного. Хороший текст достаточно короткий, но не теряет смысл.

Рабочая формула:

1. убрать мусор;
2. уточнить смысл;
3. добавить факты;
4. показать примеры;
5. выстроить структуру;
6. проверить, решает ли текст задачу читателя.

## 16. Финальный чеклист перед публикацией

Перед отправкой текста проверить:

- понятно, кто читатель;
- понятно, зачем он будет читать;
- главная мысль стоит в начале;
- структура соответствует задаче текста;
- каждый абзац раскрывает одну мысль;
- предложения не перегружены;
- есть действующие лица и действия;
- оценки заменены фактами, сценариями или примерами;
- нет канцелярита и искусственной сложности;
- термины соответствуют уровню читателя;
- после прочтения понятно, что делать дальше.

## Короткая формула

Писать так:

> Сначала польза для читателя. Потом структура. Потом ясные предложения. Потом чистка слов. Потом факты, примеры и подача.

Для рабочих текстов:

> Сначала статус и решение, потом контекст, потом факты и риски, потом детали. Не прятать вывод в середине текста. Не заменять проблему канцеляритом. Не писать «важно» — показывать, на что влияет.
